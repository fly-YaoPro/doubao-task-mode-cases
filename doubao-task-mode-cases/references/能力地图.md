## 二、内容创作组

### doubao-creative-design（图片设计统一入口）

- 模型：Seedream 4.5 默认；用户点名 seedream5.0pro 时必须保持用户原始 prompt 不变，禁止套用本 skill 任何改写规则。
- 工具：`image_gen`（0→1 文生图）/ `image_edit`（基于核心资产延展）。一致性硬规则：存在核心资产后，所有衍生物料必须用 `image_edit` 引用其 CDN URL，禁止凭文字重新 `image_gen`。
- 产物路由（命中才读对应 ref）：

| 触发 | ref |
|---|---|
| Logo/标志/标准组合/favicon | brand-logo-system |
| IP/吉祥物/三视图/表情包/角色周边 | brand-ip-character-system |
| 包装/礼盒/瓶标/刀模/SKU 变体 | brand-packaging-system |
| 品牌应用/主 KV/活动海报/宣传册/落地页/展会物料 | brand-application-system |
| 电商主图/详情页/A+页/店招/闲鱼/淘宝/京东/拼多多/亚马逊 | ecommerce-design |
| 知识科普海报/教学图/思维导图/流程图/数据图表/公式推导图 | knowledge-education-visual |
| 小红书/抖音/B站/朋友圈/公众号/社媒长图/视频封面 | media-creative |
| 其他通用图片任务 | 不读产物 ref，走主 SKILL 通用流程 |

- 路由仲裁：明确产物 > 多轮上下文 > 上传图角色 > 平台关键词 > 通用视觉任务。品牌任务先定核心资产（Logo 或 IP），再做衍生；非品牌任务不强行扩展为品牌方案。
- 门控：澄清最多 1 轮、每次 1-3 问；张数默认 1 张/对比 2-3 张/系列 3-4 张，普通任务不超 4 张；复杂任务（系列图、多物料、一致性任务）先给执行计划等确认；最多修改 1 轮。
- prompt 红线：价格/日期/功效数字只能来自用户提供，不编造；不用第三方品牌/IP/在世艺术家做风格锚；最终 prompt 不出现平台名和工具字段。
- 任务模式结合点：记住品牌色/产品参数/排版偏好，跨周保持一致；触发式批量出图（新品照片进文件夹→自动处理）。

### doubao-creative-video（生成新视频）

- 三策略路由：

| 用户目标 | 路由 ref |
|---|---|
| 口播/真人讲解/种草/测评/开箱/信息流/带货/达人视频 | ugc-talking-video-ref |
| 产品营销广告/单品种草/产品展示大片（未提信息流/口播/真人） | product-marketing-ad-video-no-storyboard-ref |
| 企业宣传片/产品功能介绍/招商/路演/客户案例/会议开场/B2B | corporate-business-video-ref |

- 硬边界：只能**生成新视频**，不能编辑/剪辑/理解已有视频（理解侧由 video-extract 互补）；短剧/剧情/影视叙事 → doubao-creative-drama。
- 硬门控：
  1. 每轮新任务必须输出参数确认摘要并等用户明确确认，历史确认不得复用；
  2. 产品/企业资料门控：命中广告/UGC/Corporate 默认先有产品图或企业资料，无资料须用户明确选"无参考直接生成"；
  3. 分镜图片门控：默认只交付分镜表，用户明确要"分镜图/关键帧图"才生成图片；
  4. 一次最多生成 2 个视频；
  5. 口播/旁白台词必须完整出现在确认摘要（UGC 默认 15 秒；Marketing 信息型 3-5 句旁白+1 句 slogan，诗性路线 2-3 句+slogan；Corporate 必须完整旁白）；
  6. 交付必须 `notify_human` 上屏视频本体。
- ref 职责分类 11 类：product / company / case / brand / script / knowledge / person / scene / style / audio / platform，多 ref 冲突有固定优先级。
- Corporate >15 秒请求先判断：多条独立 15 秒 or 长视频拆段拼接。
- 任务模式结合点：定时生成日更口播、节日营销排期；记住店铺调性、跨日主播不撞脸。

### doubao-creative-drama（短剧/微电影流水线）

- 六阶段流水线：planner（主 SKILL）→ scriptwriter → storyboard → assets → frame → prompt，每阶段完成必须用户明确确认才能读下一个 ref。
- 边界：单集 5-10 分钟以内短篇；不承接连续剧/长篇完整制作（>10 集有剧情/人设漂移风险，剧本可分批 3-5 集输出并附分集大纲）；单段视频锁定 15s。
- 剧本四段产物：剧情梗概（约 1500 字）→ 人物小传 → 主要故事场景清单 → 完整剧本（分集分场专业格式），缺一段影响下游对齐。
- 资产设定强制顺序：主角设定图→确认→配角/反派逐位确认→角色总体确认→场景全景逐场景确认→其余视角。禁止合并跳批。
- 角色形象大变（换装/受伤/年龄变化）必须先重生成该阶段角色资产图并确认，再生成后续关键帧或视频。
- 一次最多 2 个视频；图片即时输出（Markdown 图片语法）；视频必须 notify_human 上屏。
- 未做案例。短剧是抖音大赛道，潜在爆款方向；内部会调 lark-sheets 做分镜表。

### doubao-newmedia-writing（新媒体写作）

- 路由：小红书图文笔记 / 公众号文章（新闻热点/专业解读/故事纪实/认知观点/情感治愈）/ 3 分钟以内短视频分镜脚本 / 复合型创作方案。
- 硬规则：明确命中创作类型后，**飞书文档是唯一交付通道**——第一执行动作就是 `lark-cli docs +create` 建占位文档；每个类型的文档必须同时含文本内容和图片内容。
- DO NOT USE：只要标题/选题/文案/口播稿但没提平台名 → 1-2 句承接并追问，禁止编造平台或成品形态；模糊需求（"帮我写""做个爆款"）同样先澄清。
- 未做案例。

### lark-ppt（PPT 制作）

- 归飞书能力层管理，不建案例目录、不单独做案例；保留路由说明备查。
- 四工具：`ppt_write`（新增页）/ `ppt_refine`（改页）/ `ppt_delete` / `ppt_read`。
- 规则：未指定页数默认 15 页以上；字体白名单（思源黑体/宋体等中文+Poppins/Montserrat 等拉丁）；不要默认深蓝/科技蓝/米黄主色；修改图片必须先调图片工具拿链接再 refine；默认尺寸 960×540。

## 三、可视化与应用组

生命周期分流（铁律）：看懂一次、探索一次 → `doubao-visualization`；反复操作、保存状态、多人协作、持续迭代 → `doubao-app-builder`。不得只为美观升级成应用。

### doubao-visualization（对话内交互 HTML）

- 触发白名单（6 条）：显式要求可视化 / 数据图表显著增益 / 教学型动态概念（参数影响曲线轨迹）/ 步骤状态机 / 结构化知识展示 / 几何构造证明。
- 硬边界：**地图可视化绝对禁止**（Leaflet、ECharts geo/map、行政区划数据均禁）；不能替代真实附件交付（PDF/PPT/Excel 需求不可用 renderer 冒充）；取不到可靠数据时输出空态图+明确标注"未取得可靠数据"，禁止编造。
- 格式约束：`html type="renderer"` 代码块直接嵌入回复；禁 head/body/style 标签；CSS 全内联；单块 ≤300 行；高度由内容撑开（禁 100vh）；外部库优先 jsDelivr、严禁 BootCDN；移动端交互元素 ≥44×44px、正文 ≥14px。
- 有案例（cases/visualization/，9 个文件）。

### doubao-app-builder（独立网页应用）

- 单工具 `app_builder_agent`，覆盖生成/编辑/对已生成产物的问答三类任务。
- arch_type 判定：默认 `jspage`；明确需要保存业务数据/数据库/后端 → `fullstack`；3D/游戏 → `html`。**用户说"做个 XX 管理系统"但未提数据存储时，禁止直接判定，必须先对话追问**。
- query 改写红线：不替 agent 做技术选型（禁写"用 React/ECharts"等）；AI 能力诉求禁止降级为 mock；附件 URL 必须完整保留。
- 禁止越权自读产物：产物在 agent 沙箱里，读代码/读文件/查运营数据必须路由给 agent，禁止本地 ls/cat 或 web.fetch 抓产物页面，禁止凭记忆编造产物内容。
- 本地素材必须先 FileUpload/FileBatchUpload 转远程 URL；禁止用外部工具预生成图表图片传入（给原始数据+图表类型建议，由 agent 原生渲染）。
- 「按照要求修改应用 XX」前缀：user_prompt 只传"按照要求修改应用"，应用名用于匹配 app_id。
- 禁止在回复中输出应用地址（系统卡片自动展示）；Execution failed 重试必须用完全相同的 user_prompt。
- 有案例，已实测（cases/app-builder/，7 个文件）。

## 四、金融投研组

四者边界（互斥路由，写案例时先判断问题落在哪一格）：

| Skill | 管什么 | 不管什么 |
|---|---|---|
| finance-sector | 板块/概念/题材的**短期热度**：双轨打分、背离检测、10 代表股、催化归因 | 行业长期趋势、单股、财报、大盘 |
| daily-stock | **单一个股**当日/近期日报：涨跌原因、资金、新闻、技术面、预期 | 长期商业模式、财报、板块、多股、大盘 |
| market-hotspot | **市场整体/宏观**：涨跌归因、政策、跨资产联动、风险偏好 | 单股、板块、财报、荐股/买卖点 |
| earnings-analysis | **财报/业绩**深度因果：指标变动归因、超/低预期原因 | 纯股价、估值、非财报新闻 |

共同纪律：
- 数据红线：行情/资金结构化数据必须来自 `seed_finance_search`（同花顺数据库），禁止估算/记忆/新闻摘要补值；摘要 8 指标和 A 股资金流只能 seed 来源。
- 两阶段交付：先对话框完整分析 → 固定结尾句问"是否生成飞书文档" → 用户确认后复用同一底稿经 lark-doc 写飞书，禁止重新检索或改写事实。
- 免责声明固定文案，逐字输出。

各自要点：
- **finance-sector**：双轨打分（信息×行情各 1-5，综合=行情×0.55+信息×0.45）+四象限背离；最高红线——对话框只能输出经校验的 `work/<板块>_chat.md`，禁止直接输出分析草稿；脚本流水线 derive_facts→lint_analysis→generate_doc→render_chat。
- **daily-stock**：question_focus 五类（target_price/short_trade/event_catalyst/position_advice/daily_recap）；A/H 两地上市公司必须先问选哪边；量比只在收盘口径用 seed 成交量自算，盘中写"暂无"。
- **market-hotspot**：7 类问题（全市场热点/涨跌归因/宏观政策/新闻风险/跨资产/风险偏好/市场追踪）；Lens 思考框架（analysis-playbook）；飞书交付依赖 lark-doc 伴生，不可用则只交付对话框正文。
- **earnings-analysis**：报告模式（整份财报）vs 聚焦模式（1-2 个变量）；解释深度阶梯 4 层，核心论点至少推到深度 3（为什么公司不能消解）；结论分档（已证明/合理推断/单一解释），单一解释不得写成定论；中间产物目录不交付。

## 五、信息获取与本地操作组

### doubao-sentiment-tracker（舆情监控溯源）

- **PC/网页端独占**：手机端禁止触发，只提示"电脑版或网页端效果更佳"。
- 五步法：全网检索 → 重点平台深挖（微博必做，用户指定平台只是"重点关注"不缩减覆盖）→ 筛选 → 强制溯源 → 先归类总结后详情列表。
- 红线：每条舆情必须带可直接访问的原帖链接（缺一条整份不合格）；一手信源优先，二手信源必须标注原因；必须文档交付；不允许 OrganizeAgent 参与；遇登录限制必须引导用户登录，禁止换平台或放弃。
- 任务模式结合点：定时周期性舆情监控（每周/每月）+跨时间段对比。
- 有案例（cases/sentiment-tracker/）。

### browser-task（浏览器 GUI 兜底）

- 三个触发条件（先判定）：①其他 skill/搜索/API 都无法完成，必须真实浏览器 GUI 兜底；②需要登录/授权/账号内动作（点赞/收藏/评论/发布/加购）；③命中白名单站点。
- 白名单子 skill：淘宝天猫（taobao-skill）/ 微博（weibo-skill）/ 小红书（xiaohongshu-skill）。子 skill 站内规则优先于通用约束；未命中白名单走通用 GUI 流程。
- 通用约束：采集类搜索≤5 轮、最终入选 ≥10 条且去重（URL+90% 相似度）；操作类先查状态→操作→验证；时效性任务主动取最新。
- 禁止场景：纯文本/代码/已有结构化数据/API 可得/搜索已满足/无凭证登录。
- 有案例（cases/browser-task/，含采购场景）。

### computer-use（本地电脑 GUI）

- 职责：判断 GUI 任务归 MainAgent 还是 ComputerUseAgent。
- 归 ComputerUseAgent：真实本地浏览器事务（profile/cookie/扩展/原生对话框）、非浏览器 GUI（桌面应用/系统设置/安装器/文件选择器/远程桌面）、软件下载安装更新卸载（整任务）。
- 归 MainAgent：网页内容区浏览、本地文件读写、命令行、API。
- **腾讯系软件暂不支持**（微信/QQ/企业微信/腾讯会议的 GUI 一切操作），命中即停止，统一话术回复，不得绕过。

## 十、平台底座

- **doubao-cron-scheduler**：任务模式的发动机。要点：相对时间必须先 `get_current_time` 锚定；歧义时间必须给 2-3 个候选做选择题式澄清；query 必须含无人值守约束（不委派 OrganizeAgent、执行中不 askhuman）；**依赖登录态的任务创建后必须立即打开云端浏览器让用户接管登录，cookie 存云端**；不把密码/token 写进 query。任何"每天/每周自动跑"的案例底层都是它。
- **doubao-identity**：豆包会员/隐私/记忆功能问答。不做案例。
- **doubao-record**：飞书会话录音启动与查询。组合用（会议场景）。
- **skill-creator-for-task**：创建新 skill 的指南。元能力，不做案例。

## 十二、新 skill 入库流程
