# 豆包 App Builder 测试反馈文档

> 案例：多平台电商运营周报数据看板（PC桌面端场景）
> 测试场景：潮玩数码 · 四平台（淘宝/京东/拼多多/抖音）周报
> 测试日期：2026-06-23
> Skill版本：doubao-app-builder（通过 `app_builder_agent` 工具生成网页应用）

---

## 一、场景测试反馈

### 1.1 测试场景描述

**场景**：电商运营主管每周五下午坐在PC前，从四个平台后台（淘宝千牛/京东商智/拼多多商家后台/抖音电商罗盘）扒数据、做周报看板。

**具体设定**：潮玩数码（3C配件）运营主管周姐，每周五下午2点坐到自己工位双屏前——左边屏幕四个浏览器标签页对应四个平台后台，右边屏幕是Excel。本周需要汇总淘宝/京东/拼多多/抖音四个平台的GMV、订单数、转化率、客单价、退货率，并和上周做环比对比、标注异常指标。

**测试输入（模拟周五下午周姐口述数据）：**
```
帮我把这周四个平台的周报做出来。

淘宝：42800 / 1220单 / 转化8.1% / 客单35
京东：28600 / 680单 / 转化5.6% / 客单42
拼多多：15300 / 850单 / 转化4.2% / 客单18
抖音：18900 / 520单 / 转化1.8% / 客单36

上周的：淘宝38500/1150/7.8%，京东31200/750/6.1%，拼多多14100/780/3.9%，抖音22400/640/2.6%

帮我做个电脑上看的周报看板，要环比对比，异常的帮我标出来。
背景深色，像数据后台就行。
```

**Skill路由结果**：正确识别为网页应用生成 → `app_builder_agent` 调用，arch_type 默认 `jspage`

### 1.2 路由与门控行为分析

#### 一、意图识别与参数组装

Skill从用户输入中提取到的关键信号：

| 信号 | 提取值 | 处理逻辑 |
|------|--------|----------|
| 应用类型 | "周报看板"、"电脑上看" | → 网页应用生成，非3D/游戏，arch_type=jspage |
| 数据内容 | 四平台×本周/上周共40+个数据点 | → 保留原始数字，不预计算环比 |
| 图表需求 | "环比对比"+"柱状图"（用户未明确但场景隐含） | → 在user_prompt中提供原始数据+图表建议，由agent内部渲染 |
| 异常标注 | "异常的帮我标出来" | → 可补充阈值规则（转化率跌幅>20%=红、退货率>6%=橙） |
| 设计风格 | "深色"、"像数据后台" | → 风格约束写入user_prompt |
| 屏幕适配 | "电脑上看" | → 暗示宽屏布局，非移动端竖屏 |

**关键判断点**：用户输入中未提供环比数字——这是好事。如果skill在user_prompt里手动算好了环比再传给app_builder_agent，那后续"每周更新数据"时环比数字就变成硬编码的、无法自动刷新了。正确的做法是只传本周和上周原始数字，让app_builder_agent内部计算环比。

#### 二、user_prompt 组装质量（预期行为）

根据skill规则，正确组装应该：

1. **保留原始数据不预加工**：本周和上周的原始数字原样传入，不手动计算环比。例如传"淘宝本周42800 上周38500"而非"淘宝环比+11.2%"
2. **图表用原始数据**：柱状图需求提供四平台×两周的GMV原始数，由agent内部渲染
3. **不指定技术栈**：不写"用ECharts"、"用Chart.js"、"用React"
4. **不预生成图表PNG**：不外部调用Python/matplotlib生成图片
5. **风格约束具体化**：不仅是"深色"，而是给出参考风格（如"类似Grafana的数据后台风格"）
6. **异常规则可执行**：不写模糊的"标出异常的"，而是转化为可量化的规则（"转化率环比跌幅>20%标红"）

#### 三、看板与PC场景的适配分析

这是本案例区别于一般app-builder测试的核心维度——产品是否真正符合"坐在双屏电脑前深度使用"的场景，而非被迫适应了手机端的设计惯性。

**PC端适配的关键判定点：**

| 维度 | 移动端习惯 | PC端正确做法 | 对本案例的影响 |
|------|-----------|-------------|--------------|
| 布局 | 单列堆叠、卡片垂直排列 | 多栏并排、充分利用横向空间 | PC宽屏下能否同时看到"数据总览卡片+对比表格+异常标注区"三栏 |
| 信息密度 | 一屏只看3-4个数字 | 一屏承载20+数据点+图表 | 四平台40+数据点能否在一个视图内呈现而不需要滚动 |
| 图表尺寸 | 小尺寸，占半屏 | 充分放大，柱状图横轴标签不重叠 | 四平台并排柱状图在手机端会挤成一团，PC端应舒展 |
| 交互暗示 | 触摸大按钮、滑动卡片 | 鼠标hover、点击筛选、桌面端表格排序 | 表格列头是否支持点击排序（如按GMV降序排列） |
| 字体层级 | 大号数字、留白多 | 合理字号、紧凑但可读 | 总GMV数字应该大但不应该占半屏 |

**场景真实性风险**：如果app_builder_agent默认的渲染策略偏向移动端（很多网页生成工具会默认mobile-first），产出的看板可能是一列卡片纵向排列，装在PC宽屏上感觉空旷且信息密度不够。这是本案例测试需要重点盯住的。

#### 四、跨周模板化的能力分析

这是任务模式的核心价值——不是"帮我做一次"，而是"以后每周都用这个模板"。

**技术层面**：app_builder生成的是静态页面。跨周复用的关键在于——用户输入新数据后，skill能否用"按照要求修改应用"指令触发app_builder_agent重新渲染页面、替换数据、重新计算环比和异常标注。这要求：

1. 初始看板的数据结构不能是"硬编码在HTML里的死数字"
2. app_builder_agent在收到"按照要求修改应用"指令时，能定位到数据部分并替换
3. 替换后不会破坏页面的布局、配色、图表结构

**如果跨周更新失败**——比如app_builder_agent每次"修改"都是重新生成一个页面，布局变了、配色变了——那任务模式的"以后每周用"就名存实亡。用户还不如每次从头生成。这是本案例最大的验证风险点。

### 1.3 区别于普通模式的突出用法

| 维度 | 普通对话模式 | 任务模式 + App Builder |
|------|-------------|----------------------|
| **交互方式** | 每次从零描述需求："帮我做一个周报看板，淘宝京东拼多多抖音……"——下周再说一遍 | 上下文延续：skill记住了"潮玩数码的四平台周报模板"。下周只需输入新数据，看板自动套模板、算环比、标异常 |
| **数据处理** | 对话可能帮你手动算环比："淘宝这周42800上周38500，涨了11.2%"——下次你忘了上周数字，环比就没了 | 原始数据持久化在任务会话上下文中。下周输入本周新数据时，上周数据作为参考基线保留，环比自动重算 |
| **模板一致性** | 每次生成风格可能漂移——上周是深蓝背景三栏布局，这周可能变成浅色背景单列卡片 | 模板锚定：同一个app_id下，布局、配色、图表类型、异常规则全部保持。只有数据被替换 |
| **异常检测规则** | 每次都要重新说"转化率跌了标红" | 规则持久化：首次设定"转化率跌幅>20%标红、退货率>6%标橙"后，每周自动执行，不需要重说 |
| **跨周趋势** | 对话结束后不记得上周数据，无法自动显示"连续两周下跌"之类的趋势警告 | 任务模式可在上下文中维护多周历史数据，当某指标连续两周恶化时自动升级警告级别 |
| **PC场景适配** | 对话是纯文本交互，不感知设备。可能在手机上聊着聊着生成了一个看板，但在手机上看周报细节很痛苦 | 任务模式可绑定PC场景：每周五下午2点在桌面端触发提醒，生成PC宽屏布局的看板，直接在工作电脑上打开 |

**最核心的差异一句话**：
> 普通对话 = "帮我做一次周报"；任务模式 + App Builder = "以后每周五你不用管了——到点提醒你输入数据、自动套上周模板、环比自动算、异常自动标。你做运营分析，AI做表格"

### 1.4 测试中暴露的能力边界与关注点

| # | 关注点 | 预期行为 | 潜在风险 | 严重程度 |
|---|--------|----------|----------|----------|
| 1 | **环比自动计算** | app_builder_agent在接到本周和上周原始数字后，应在页面内自动计算环比涨跌并展示 | 如果看板上的环比数字是用户手动算好提供的（而非agent自己算的），说明数据是硬编码的——"下周更新"时环比不会自动刷新 | 🔴 致命——直接决定"以后每周用"是否成立 |
| 2 | **图表原生渲染** | 柱状图/折线图由app_builder_agent内部渲染为交互式图表（如SVG/Canvas），而非外部生成的静态PNG | 如果是PNG贴图，后续没有任何办法更新图表数据——除非重新生成一张新PNG替换。但那样每次修改都依赖外部工具，破坏了app_builder的闭环 | 🔴 致命——决定看板是"活"的还是"死的" |
| 3 | **PC宽屏布局** | 看板应利用横向空间，多栏并排。数据卡片、表格、异常标注区同时可见 | 如果布局是手机端单列堆叠（卡片一个接一个竖排），在1920宽屏上右侧大量空白、信息密度不够 | 🟡 重要——虽然数据正确但场景体验失真 |
| 4 | **数据结构可替换性** | agent在设计看板时，数据部分应该是可替换的结构（变量/配置驱动），而非硬编码在HTML标签里 | 如果首次生成时数据被写死在页面源码里，后续"修改应用=更新数据"时会变成全文替换——连布局都可能被破坏 | 🔴 致命——决定模板化是否成立 |
| 5 | **编辑指令的精准度** | "按照要求修改应用+应用名称"应只修改用户指定的内容（更新数据、标红某行），不动布局 | 常见问题：agent在"修改"时理解偏差，把整个页面重新生成了一遍，布局变了、配色变了、图表类型变了 | 🟡 重要——影响跨周一致性 |
| 6 | **异常标注规则的执行** | 用户说"异常的帮我标出来"→agent应量化规则并执行（如转化率跌幅>20%标红） | 如果用户没有给定具体阈值，agent可能不知道什么叫"异常"——把所有跌的都标红（过度标注）或者一个都不标（欠标注） | 🟡 重要——影响看板的实用性 |
| 7 | **感知/问答的深度** | 针对已生成的看板问"分析一下抖音为什么转化率跌"，agent能否基于看板数据给出分析 | 如果agent只能"读页面上的文字"而无法做数据推理，回答可能是"页面上显示抖音转化率1.8%，上周2.6%"——复述数据而非分析 | 🟢 锦上添花——有了更好 |
| 8 | **数据输入的便利性** | 理想的跨周更新流程：用户口述"这周淘宝45000、京东29000..."→agent自动匹配到对应平台行→替换数据 | 如果每次更新都需要用户先精确描述"修改淘宝GMV为45000、修改淘宝订单数为1280..."逐字段指定，更新过程可能比手动做Excel还繁琐 | 🟡 重要——影响实际使用体验 |

## 二、场景发散

### 2.1 直接衍生场景（改数据源，不改看板模式）

同一个"多源数据→周期性看板"链路，直接迁移到以下PC场景：

| 场景 | 人群规模 | PC端典型时刻 | 与电商周报的差异 | 传播钩子 |
|------|----------|-------------|-----------------|----------|
| **新媒体运营周报** | 全国约300万新媒体从业者 | 每周一上午，打开公众号后台+小红书创作中心+抖音创作者平台+B站创作中心，四个页面来回切，手动汇总上周阅读量/点赞/收藏/评论/粉丝增长 | 数据维度不同（内容指标而非GMV），但看板结构相同——多平台对比+环比+趋势+爆款标注 | "周一上午别人在四个后台切来切去抄数据，AI已经把上周内容周报生成好了" |
| **广告投放优化师日报/周报** | 全国约50万投放从业者 | 每天上午9点，打开巨量引擎+腾讯广告+百度营销三个后台，记录每个计划的消耗、ROI、CPA、CTR。一周下来几十个计划的数据，周报要做2小时 | 比电商多一个维度——计划/账户级别下钻。看板需要支持"总览→分平台→分计划"的层级 | "别人在Excel里手动拉ROI公式，AI把每个计划的投放效率自动排好序了" |
| **连锁门店区域经理周报** | 全国连锁门店区域经理约20万人 | 每周五下午，从公司ERP导出辖区15家门店的营业额、客流、客单价、人效、好评率。Excel里做15行×8列的数据表，手动排名、手动标红垫底门店 | 数据来源从"多平台后台"变成"单一ERP导出"，看板重点是排名和垫底标注 | "15家门店的数据，AI自动给你排好名、标出垫底的、写好异常说明" |
| **跨境电商多站点周报** | 全国约100万跨境卖家 | 每周一上午，打开Amazon Seller Central(美国站+欧洲站+日本站)、速卖通、Shopee、Lazada。各站点货币不同、时区不同，还要换算汇率 | 增加了汇率换算、时区对齐、多语言产品名等复杂度。看板需要在同屏展示不同币种（可自动换算为人民币） | "六个国家六个后台，AI帮你统一换算成人民币、自动标出哪个站点这周拖后腿了" |

### 2.2 跨行业衍生（保留"周期性看板+PC场景"，换信息输入方式）

| 场景 | 原始模式 | 变形后 | 变形要点 |
|------|----------|--------|----------|
| **工程项目经理周报** | 电商GMV数据→看板 | 项目进度/工时/预算消耗→看板 | 输入从"销售额/订单/转化率"变成"完成百分比/消耗工时/预算余额"。异常标注从"转化率跌"变成"进度滞后>20%""预算超支>10%" |
| **客服团队周报** | 电商数据→看板 | 工单量/满意度/首次解决率/平均响应时长→看板 | 数据来源从平台后台变成工单系统（如智齿/网易七鱼/Udesk）。看板重点是"哪个客服评分最低""哪类问题工单暴涨" |
| **HR招聘周报** | 电商数据→看板 | 各渠道简历量/面试到场率/offer接受率/到岗周期→看板 | 数据来源从电商后台变成招聘平台（BOSS直聘/猎聘/拉勾）后台。看板重点是"哪个渠道性价比最高""哪个岗位卡住了" |
| **供应链采购经理周报** | 电商数据→看板 | 各供应商报价/交货准时率/次品率/库存周转天数→看板 | 看板重点是"哪家供应商涨价了""哪个SKU库存告急"。异常标注规则更复杂——次品率>3%标红、交货延迟>2天标红 |

### 2.3 模式组合创新（App Builder × 任务模式其他能力）

**组合一：周报看板 × 浏览器操作 × 自动数据抓取**
```
场景：进阶版——运营不再手动输入数据
任务模式设定：每周五下午2点 → 自动在PC浏览器中打开已登录的
淘宝千牛/京东商智/拼多多后台/抖音罗盘四个标签页 → 
自动抓取各平台本周核心指标 → 填充周报看板模板 → 推送→ 人工审核确认后发送
```
这是从"对话型数据输入"升级到"全自动数据采集"。PC场景的优势：浏览器标签页常开、已登录状态持久化、数据就在页面上可见。

**组合二：周报看板 × 竞品监控 × 对比分析**
```
场景：电商运营的周报不止看自己，还要看竞品
任务模式设定：每周一上午 → 自动抓取3家竞品店铺的公开数据
（销量排序Top10商品、价格变动、新品上架、活动banner）→ 
和自家数据并排显示 → 标注"竞品A这周上了3个新品""竞品B降价了15%""竞品C参加了平台满减活动"→ 
生成"竞品动态+自家表现"的综合周报看板
```

**组合三：周报看板 × 多角色分发**
```
场景：同一份底层数据，不同人看不同维度
- 运营主管 → GMV/转化率/爆款排行视角
- 客服主管 → 退货率/差评归因/响应速度视角
- 仓管 → 库存周转/滞销预警/补货建议视角
- 老板 → 利润率/现金流/同比增长视角

任务模式：周五下午生成一份主看板 → 自动拆分为四个角色版本 → 分别推送
```

### 2.4 不适合的场景（做减法）

| 场景 | 否决原因 | 对应筛子 |
|------|----------|----------|
| 个人记账/消费分析看板 | 纯个人生活，非生产力定位 | 第0筛 |
| "帮我列几个数字做个表" | 单步任务，对话完全能独立完成。没有周期性复用需求 | 第一筛 |
| 实时大屏监控（每秒刷新数据） | app_builder生成的是静态页面而非实时数据流应用。WebSocket/轮询等实时机制超出了当前生成能力 | 技术边界 |
| 需要接入API的外部数据源 | 如果数据源只能通过API获取（需要开发对接），而非用户在浏览器页面上可见的——app_builder无法直接接入 | 技术边界 |
| 代记账/财务报表 | 会计准则合规要求，AI生成表格容易出错且无法追溯 | 合规红线 |

## 三、总结判断

### 3.1 案例可行性

案例 B"多平台电商运营周报数据看板"在能力模式"多源数据→周期性分析看板"中**可以成立**。它与能力模式二（监控→推送消息）和已被提出的"数据→看板"类案例的本质差异在于：

1. **不只是"展示数据"，而是"周期性分析"**：环比对比、异常标注、趋势判断——这些都是分析动作，不是简单的数据罗列
2. **PC场景的独占性**：周报的物理场景永远是一张桌子+一台电脑+双屏——这是任务模式在PC端最自然的切入点
3. **跨周模板化的能力验证**：如果app_builder能做到"同一模板、每周只换数据、布局和规则全保持"，就证明了任务模式在"应用生成"这个能力支柱上成立

### 3.2 核心风险（如果测试失败，最可能败在哪）

按概率排序：

1. **跨周更新不一致**（最高风险）——app_builder_agent在接收"按照要求修改应用"指令更新数据时，可能会重生成整个页面而非精准替换数据部分，导致布局/配色/图表结构漂移
2. **PC宽屏适配缺失**（较高风险）——agent默认生成移动端优先的布局，在PC上显得空旷稀疏、信息密度不够
3. **数据硬编码**（中风险）——周报数据被写死在页面源码里，后续更新必须依赖"代码级修改"
4. **异常规则模糊执行**（低风险）——"异常的帮我标出来"这种模糊指令，agent无法转化为可执行的量化规则，需要用户在prompt里补充阈值

### 3.3 建议优先推进

如果这个案例在测试中跑通（跨周模板化+PC布局+原生图表+自动环比），它将是一个"能力模式×PC场景"的标杆案例——证明任务模式不只是移动端"推送一条消息"的场景，而是能承载PC端深度工作流。

---

## 四、PC场景的通用性判断

本案例验证成功后，可以确立一条新的案例设计原则：

> **凡是"坐电脑前、开多个后台、定期做的一件事"——就是App Builder任务模式的天然场景。**

不需要用户会写代码。不需要用户有数据思维。只需要用户坐在电脑前、说出他每周五下午在Excel里手动干的那些事——app_builder负责生成承载这件事的工具，任务模式负责让它每周自动跑。
