# 豆包 任务模式 — 达人选题案例手册

## 一、项目背景与最新定位

### 1.1 阶段变化

**原阶段（2026-06-18）**：为豆包App端办公任务模式做达人推广。仅限移动端，三个能力支撑中"Skill调用"未上线暂不投放。

**当前阶段（2026-06-23）**：

- **端侧扩展**：从纯App端扩展到PC端，且PC端作为生产力场景的主推阵地。PC场景天然适合深度工作流（多平台后台操作、数据看板、周报分析等），任务模式在PC端的差异化比移动端更强
- **Skill生态上线**：豆包现已支持安装各类Skill（如doubao-creative-video创意视频、doubao-app-builder网页应用生成等），支撑三（云端调用各类Skill）已从"未上线"变为"已上线"。推广逻辑从"豆包办公模式"转向**"豆包Skill生态"**——对标Codex/Codex的skill安装+任务模式编排
- **豆包独特优势**：多模态模型（Seedream图像生成、Seedance视频生成国内领先）、中国本地可用（无需翻墙）、交互便利（桌面端+App端打通）

### 1.2 推广定位重述

**原有定位（App端阶段）**：任务模式 = 一只"云端的手"。能力上限 = 浏览器的能力上限。

**当前定位（Skill生态阶段）**：任务模式 = 一个能调用各种Skill的"云端代理人"。他能安装skill、定时调度skill、跨工具编排skill——就像一个不会累的实习生，你告诉他"以后这类活你帮我盯着、到点自己用对应的skill搞定"。能力上限 = 已安装Skill的能力并集 × 任务模式的定时/调度/跨工具编排能力。

**核心区别**：普通对话是"此刻给我一个答案或用一个skill"；任务模式是"以后这件事你替我盯着、到点自己调度对应的skill办——我不用每次重复交代背景、不用记得上次的参数、不用手动跨工具串联"。

### 1.3 与 Codex / Codex 的对标逻辑

豆包的Skill生态推广逻辑，与Codex的slash commands/workflows、Codex的skills/automations高度相似。达人不需了解底层差异，但案例设计者需要理解对标关系：

| 维度 | Codex | Codex | 豆包任务模式 |
|------|------------|-------|-------------|
| Skill安装 | 通过AGENTS.md配置 | 通过.codex/skills安装 | 豆包内置Skill市场 |
| 任务调度 | 手动触发或cron | 定时+事件触发 | 任务模式=定时+条件触发 |
| 跨工具编排 | MCP工具链 | MCP+Skills | 浏览器+搜索+Skill调用 |
| 多模态能力 | 文本为主 | 文本+图像理解 | Seedream(图)+Seedance(视频)原生多模态 |
| 本地可用性 | 需网络+账号 | 需网络+账号 | **中国本地无需翻墙** |

**一句话差异化**：Codex/Codex胜在开发者生态和代码能力，豆包胜在多模态（图+视频原生生成）+中国本地可用+交互便利。做案例时强调"会用手机/电脑就能用"，而不是"需要配环境写配置"。

### 1.4 PC端场景的方法论补充

PC场景和App场景的本质差异，影响案例设计和筛选标准：

| 维度 | App端 | PC端 |
|------|-------|------|
| 物理场景 | 碎片时间、单手操作、看一眼就走 | 深度工作时间、双屏、多标签页、坐1-2小时 |
| 信息密度 | 低——推送一条消息、一个数字 | 高——看板、表格、图表、多维度同时对比 |
| 用户动作 | 接收消息→判断→简单回复 | 打开看板→分析数据→做决策→写报告 |
| 数据输入 | 口述/拍照 | 复制粘贴/上传文件/从浏览器标签页抓取 |
| Skill偏好 | 轻量Skill（口播视频、简单生成） | 重量Skill（数据看板、长视频、应用生成） |
| 推广节奏 | 先做移动端已有认知的场景 | PC端场景需要建立新认知——"AI不只是聊天，是坐在你电脑旁边的助手" |

**PC场景新增筛选维度**（叠加在原五道硬筛之上）：
- **PC-1**: 场景的物理动作是否天然发生在电脑前？（如"做周报"天然PC，"查天气"天然移动端）
- **PC-2**: 信息密度是否超出手机屏幕的承载能力？（如4平台×10指标=40个数据点，手机看不全）
- **PC-3**: 是否涉及多窗口/多标签页操作？（如同时开4个平台后台）

命中2个以上 → 优先作为PC端案例推广；命中0-1个 → 保持移动端或双端兼容

## 二、核心认知

### 2.1 vs 普通对话

| 维度 | 普通 AI 对话 | 任务模式（对话做不到的） |
|------|------------|---------------------------|
| 时间 | 你问一次，它答一次 | 定时自己跑，在未来某刻主动行动 |
| 主动性 | 你喂数据它才动 | 持续盯着变化，满足条件主动推送 |
| 跨工具 | 只能输出文字 | 自己开浏览器/登录网页版后台连续操作 |
| Skill调用 | 单次调用一个Skill | 定时或条件触发Skill、跨Skill编排 |
| 上下文 | 对话结束=状态丢失 | 跨周/跨月持久化——记住你的模板、规则、偏好 |
| 能力边界 | 你能干的它帮你快点干 | 替你干你自己干不了/嫌麻烦的电脑级活 + 替你记住你记不住的事 |

### 2.2 四大能力支撑（更新）

1. **支撑一: 定时自动 & 持续监控** ——"以后这件事你替我盯着、到点自己办"（已上线，案例库主战场）
2. **支撑二: 云端电脑 / 云端的手** ——"替你干电脑、网页才能干的活"（已上线，最具传播张力）
3. **支撑三: 云端调用各类 Skill** ——**已上线（2026.06）**，豆包现已支持安装doubao-creative-video/doubao-app-builder等Skill。任务模式可定时或条件触发Skill执行。**这是当前阶段的核心增量**
4. **支撑四展望: 手机操控电脑** ——终极形态，上线时间待定，暂不投放

### 2.3 内容爆破逻辑

三个点: 认知冲击(前3秒"AI居然能干这个！") → 代入感(中间15秒"如果我也有…") → 行动冲动(最后3秒"现在就想试试")

拍法公式: 展现痛苦(10-15秒，越真实越好) → AI一句话搞定(5秒，视觉反差) → 展示完整成果(10秒) → "你也可以"

## 三、推广沟通口径

### 3.1 推荐讲法

**原有核心三条**（移动端阶段）：定时（到点自己执行）/ 主动（不需要你来问，满足条件主动推）/ 云端代操作（替你操作浏览器/网页）

**新增核心叙事**（Skill生态阶段）：
- **"豆包能安装各种Skill，任务模式帮你调度它们"**——把豆包讲成一个能装App的AI
- **"以前你要自己学用各种工具，现在装个Skill、告诉任务模式什么时候跑就行"**——降低认知门槛
- **"手机端轻量操作，PC端深度干活"**——端侧互补而非替代

### 3.2 不建議讲法

- "替你做决策、做判断" ——定位是助手不是替代决策
- "生活化" ——本轮不做
- "比Codex/Codex更强" ——不搞横向对比，强调豆包的独特优势：多模态+本土可用+零门槛
- "开发者/程序员才能用" ——Skill安装和任务模式都是对话式操作，不需要技术背景

## 四、选题筛选框架（五道硬筛 + PC场景三维）

### 4.1 五道硬筛（继承原框架，微调第四筛）

### 第 0 筛: 生产力定位 (一票否决)
只做生产力/职业/经营/搞钱。禁止: 追星、看球、旅游、母婴、娱乐。

### 第一筛: 普通对话能做吗？（看粒度，非一刀切）
独立任务对话能干淨利落完成 → 不适合作为主卖点。但作为多步链条中的一环 → 反而是任务模式优势。
示例: "生图"对话能做 → 单独不推荐。但"拍照→去背景→加文字→排版→加QR→输出"五步链对话串不起来 → 任务模式优势。

### 第二筛: 现有手机 App 能做吗？
独立 App 能完成同样任务 → 否决。App 能做但需要人工触发/不能定时/不能跨App/不能跨工具编排 → 任务模式仍有优势。

### 第三筛: 大众真的在乎吗？
不在乎 → 选题不成立。必须是真金白银的痛点。

### 第四筛: 信息/数据可获取吗？（三档制，非一刀切）
- 安全区: 纯公开网站无需登录
- 可探讨区: 有正经桌面端网页后台，用户登录一次后会话可能持久化
- 禁区: 纯移动端无桌面后台 / 数据无法提取 / 涉及支付交易资金操作

**PC场景下第四筛补充**: PC端的"可探讨区"范围比移动端大——因为PC浏览器天然保持登录态、多标签页常开、数据在页面上可见。很多在移动端属于"禁区"的场景（如多平台后台数据对比），在PC端降级为"可探讨区"。

### 第五筛: 信息源质量 ——必须优先官方一手源
当案例涉及从网页采集信息，必须要求优先直达官方一手源，不能停留在二手自媒体转述/推测。如果官方源搜不到 → 标注"未找到官方公告"，不自推断。

### 4.2 PC场景三维（新增，叠加在五道硬筛之上）

用于判断一个选题是否更适合PC端推广（而非移动端）：
- **PC-1 物理场景**: 这件事的天然工作姿势是坐在电脑前还是拿着手机？(如"做周报"天然PC，"等推送通知"天然移动端)
- **PC-2 信息密度**: 需要同时看多少数据点？超过15个数据点时手机屏幕无法有效承载
- **PC-3 操作复杂度**: 是否需要频繁切换窗口/标签页、或需要键盘输入大量文字？

命中2个以上 → 优先作为PC端案例推广；命中0-1个 → 保持移动端或双端兼容

## 五、案例设计方法论

### 5.1 选能力模式，不选行业（核心原则不变）

早期设计案例时犯的错误：按"哪个行业还没覆盖"来堆案例（律师/药监/学术/财务/猎头……），导致大量同质化"盯→推"案例。正确做法是：
**一个能力模式只保留一个最典型案例。达人不需要 20 个"盯政府网站"的变体。**

### 每个确认案例需要具备
1. **完整的叙事弧**: 30 秒短视频能拍出来——有痛苦、有转折、有结果
2. **真实的画面感**: 不是"盯→推"的冷冰冰机制，而是"一个人在做一件事然后 AI 帮他搞定了"
3. **可量化的利益**: "省了多少钱？""少亏了多少？""没被罚多少？"
4. **可极速传播的一句话**: 听到的人能复述给别人

### 5.2 案例选择优先级
1. 有实测链接的 → 优先
2. 有明确的人/痛/解决叙事的 → 优先
3. 同能力模式下找最具传播力的 → 保留
4. 只是"能过筛"但没有画面没有实测的 → 不保留

### 5.3 Skill选型原则（新增）

当案例涉及调用豆包Skill时，遵循以下选型逻辑：
1. **先识别案例需要的能力动作**（如"生成口播视频""生成数据看板"），再匹配对应的Skill
2. **一个案例只主推一个Skill的深度使用**——案例的目的是展示"这个Skill+任务模式能干什么"，不是展示Skill列表
3. **优先选有完整测试反馈的Skill**（实测过的 > 只读过文档的 > 只听过名字的）
4. **Skill的复杂度和案例的传播力成反比**——越复杂的Skill（如corporate商务视频需要四类必填信息），案例叙事越难压缩到30秒

### 5.4 可视化与网页设计门槛（新增）

当案例涉及 `doubao-visualization`、`doubao-app-builder`、HTML 页面、数据看板、详情页或活动页时，必须先读取并执行 [可视化与网页设计质量规范](references/%E5%8F%AF%E8%A7%86%E5%8C%96%E4%B8%8E%E7%BD%91%E9%A1%B5%E8%AE%BE%E8%AE%A1%E8%B4%A8%E9%87%8F%E8%A7%84%E8%8C%83.md)。

核心要求：

1. 开案前联网搜索最近12个月内的同类优秀设计，保留3-5个有效参考；
2. 先提出一个与内容强相关的核心视觉概念，再写布局、图形、动效和配色；
3. 禁止用“高级感、科技感、渐变、玻璃拟态”等空泛风格词替代设计；
4. 先按使用生命周期选 Skill：用户“看懂一次、探索一次”优先 `visualization`；用户需要反复操作、保存状态、多人协作或持续迭代时才选 `app-builder`；不得只为画面更美观而升级成应用；
5. 提示词完整度会显著影响视觉差距：简单描述下 `app-builder` 的默认补全和设计下限通常更高；专家级提示词可让 `visualization` 在单页呈现上接近应用，但不能补齐持久化、协作和业务流程能力；
6. 实测桌面端与移动端，达到视觉规范80分入库线后才能标记为优质案例。

### 5.5 提示词双模式（默认规则）

案例提示词保留两种模式：

1. **普通用户共创模式（默认）**：先用第一人称讲具体痛点，不提前指定Skill、图表、布局或功能；让豆包依次理解问题、提出方案、补全设计、自己写出详细执行提示词，用户确认后再调用Skill。
2. **专家直出模式（按需）**：由案例设计者直接提供完整提示词，用于用户明确要求一段话执行、严格回归测试、高风险口径核对或成熟方案落地。

用户未指定模式时，一律使用普通用户共创模式。不要因为案例设计者会写长提示词，就默认把完整答案提前喂给豆包。

新案例默认交付“五轮共创对话 + 独立验收真值”；专家长提示词只在命中条件时追加。历史案例中的长提示词保留为专家模式和回归基线。

涉及两种Skill同题比较时，前四轮对话和附件必须完全一致，最后一轮只替换Skill名称。详细流程见[提示词双模式与普通用户共创流程](references/%E6%8F%90%E7%A4%BA%E8%AF%8D%E5%8F%8C%E6%A8%A1%E5%BC%8F%E4%B8%8E%E6%99%AE%E9%80%9A%E7%94%A8%E6%88%B7%E5%85%B1%E5%88%9B%E6%B5%81%E7%A8%8B.md)。

### 5.6 视觉创作类提示词质量门控（对接 yaoshi-doubao-visual-lab）

当案例涉及生图/生视频（Seedream/Seedance），需要引入 yaoshi-doubao-visual-lab 作为提示词质量标尺。但**不是把 visual-lab 当成提示词生成器**——客户要求达人用大白话引导豆包自己写到那个水平。

两个 skill 的分工：
- **visual-lab** = 标尺。定义什么样的提示词算合格（八变量、诊断顺序、自由度预算、逐镜结构）
- **doubao-task-mode-cases** = 翻译层。设计达人用大白话怎么说，豆包才能写出合格提示词

案例设计者写五轮脚本时，对照 [视觉创作质量门控](references/视觉创作质量门控.md) 中的两张表：
1. 质量门控速查表：visual-lab 的每层结构 → 用户大白话怎么说 → 豆包输出必须包含什么 → 达人怎么简易自检
2. 五轮门控嵌入点：每轮对话中应自然嵌入哪些质量门控

案例交付时，在脚本中嵌入**第四轮质量自检**（达人内心旁白，不对豆包说）——告诉看案例的达人：你不需要懂 prompt 工程，只需要查 6 点就能判断豆包写得好不好。此环节不喂给豆包。

注意：visual-lab 的素材库（media-manifest.json，346 图+77 视频）当前未落地。在素材库可用之前，案例中的参考片匹配/正负样本对照能力暂不可用，但不影响方法论层面的门控。

## 六、常见错误（更新）

1. 按行业堆案例而非按能力模式选案例 → 导致大量同质化变体
2. 第四筛一刀切"登录就否决" → 应该是三档制评估
3. 没有实测验证就放进案例库 → 需基于桌面实测记录筛选
4. 案例没有完整叙事弧 → 30 秒视频拍不出
5. **新增**: Skill案例变成"Skill功能介绍"而非"任务模式×Skill的组合叙事" → 案例的重点是"任务模式怎么让这个Skill发挥出对话做不到的价值"，不是"这个Skill有哪几个功能"
6. **新增**: PC端案例用了移动端的叙事方式 → PC场景的传播视频需要展示屏幕操作、键盘输入、多窗口切换等桌面端特有的视觉语言
7. **新增**: 忽视Skill自身的能力边界 → 每个Skill有明确的路由策略和门控规则，案例设计必须先理解Skill能做什么、不能做什么
8. **新增**: 可视化或网页只写“简洁高级、有科技感” → 没有近期参考、内容专属视觉概念和响应式实测，极易生成通用AI仪表盘
9. **新增**: 默认把专家级长提示词直接交给豆包 → 普通用户无法代入，也测不到豆包能否自己理解痛点和补全方案；除非明确命中专家直出模式，否则应走五轮共创
10. **新增**: 因为应用生成默认更美观，就把一次性展示需求全部路由给 `app-builder` → 生成时间更长、功能膨胀，也掩盖了 `visualization` 轻量快速的价值；“同提示词横测”只能比较默认补全能力，不能直接代表两个 Skill 的能力上限
11. **新增**: 视觉创作案例不嵌入质量门控 → 达人看完案例仍不会判断豆包写得好不好；必须在脚本中嵌入第四轮质量自检，且 visual-lab 是标尺而非替代，不得用它直接生成完整提示词取代五轮共创

***

## 七、铁律（更新）

- 独立生图/写文案/表格/总结/规划 → 不推荐；多步链条一环/作为Skill内部封装流程的一部分 → 可成立
- 生活化/娱乐 → 第0筛否决
- 纯移动端无桌面后台 → 第四筛禁区（PC端可探讨区更大，参考PC场景补充说明）
- 搜索优先直达官方一手源 → 第五筛要求
- 投资理财 → 守红线 | 代操作资金 → 用户确认
- 一个能力模式只保留一个最典型案例，不堆同质化变体
- **新增**: Skill案例必须实测或已完成深度分析才能入库，禁止只读文档就放案例 → 不实测你不知道Skill的门控在哪里、路由判断对不对、脚本质量到底怎样
- **新增**: PC端案例必须标注PC-1/2/3命中情况，优先推PC-3全中的案例
- **新增**: 可视化和网页案例必须先做近期设计调研，再按视觉质量规范写提示词和验收；漂亮但难用、复杂但没信息价值，同样不得入库
- **新增**: 可视化与应用生成按生命周期分流——看完即走用 `visualization`，反复操作/保存/协作/迭代用 `app-builder`；没有持久使用需求，不得只为美观选择应用生成
- **新增**: 用户未指定提示词模式时，默认普通用户共创；先讲痛点，后谈方案，最后才调用Skill
- **新增**: 预期真值和验收标准可以由案例设计者提前准备，但不得在共创模式第一轮泄露给豆包（视觉创作类同理——独立检测真值表不喂给豆包）
- **新增**: 视觉创作类案例的提示词质量用 visual-lab 方法论做标尺，但对话脚本用大白话——visual-lab 的术语不出现在用户指令里
- **新增**: 案例设计者 vs 达人的输出要区分：案例文件是给设计者看的（可以引用 visual-lab 框架），达人最终拿到的只是五轮对话脚本+质量自检——不需要他懂方法论
