 # 视觉创作质量门控（对接 yaoshi-doubao-visual-lab）
 
 > 本文档是 doubao-task-mode-cases 与 yaoshi-doubao-visual-lab 之间的桥。
 > visual-lab 提供"什么样的提示词算合格"的标准，本文档提供"达人用大白话怎么让豆包写出合格提示词"的翻译层。
 
 ## 一、职责边界
 
 | | yaoshi-doubao-visual-lab | doubao-task-mode-cases |
 |---|---|---|
 | **角色** | 提示词质量标尺（方法论） | 对话引导方案（执行） |
 | **给谁用** | 案例设计者（你） | 达人 + 豆包 |
 | **输出** | 合格 prompt 长什么样 | 怎么说大白话让豆包写到那个水平 |
 | **不该做** | 直接生成完整 prompt 给达人 | 复述全套方法论 |
 
 **协作原则**：doubao-task 引用 visual-lab 作为验收标准，不把它复述成第二份方法论。
 
 ## 二、质量门控速查表
 
 案例设计者写五轮脚本时，对照此表确保每轮用户指令里**自然嵌入了质量门控**。
 
 | visual-lab 结构层 | 用户大白话（怎么说） | 豆包输出必须包含 | 达人简易自检（看什么） |
 |---|---|---|---|
 | **八变量对齐** | "这个视频在哪用？给谁看？屏幕是横的还是竖的？" | 任务类型、画幅比例、目标平台、受众、核心信息 | 三句话能说清：在哪播+给谁看+让人记住什么 |
 | **绝对锁定** | "绝对不能画错的是什么？把原文/原理的具体事实列出来" | 逐条引用原文/原理中的事实，标注"不可偏离" | 数一下锁定了几个事实——少于 3 个说明太松 |
 | **创意母题** | "每个镜头让人记住一句话——是什么？" | 每镜一句统领性概念，不是关键词堆砌 | 用这镜母题去问路人能不能秒懂——不能就是废的 |
 | **诊断顺序** | "之前失败的原因是什么？不是笼统的'不好看'，是具体哪个层级出问题了" | 至少命中一个：任务错位 / 产品逻辑错误 / 层级错误 / 路径错误 | 诊断是否具体——"不分镜"比"不好看"好；"箭头方向没锁"比"不够专业"好 |
 | **构图骨架** | "画面里谁在前谁在后？东西摆在哪？有留白吗？" | 主体层级、构图骨架（中心/对角/S形等）、前后景关系、文字安全区 | 闭眼能说出大概布局——不能就是太泛 |
 | **材质光影** | "画面是什么质感的？光从哪来？颜色别全靠AI猜" | 至少一组材质反差 + 主光源方向 + 色彩约束 | 是否说了"光从哪来"——没说等于没写 |
 | **文字策略** | "哪些字让AI写？哪些字后期做？大标题和小字分别怎么处理？" | 明确分界线：哪些给模型、哪些后期 | 是否说了"小字、参数、长文后期处理"——没说等于默认全给AI（会翻车） |
 | **禁止项** | "最容易跑偏的是什么？别写'不要丑'——写具体的" | 每镜至少 2 条具体禁止项，直接针对本镜风险 | 逐条检查：有没有"不要丑""不要不好看"这种废话？有就不合格 |
 | **分镜与节奏** | "这一段该拆成几个镜头？每镜几秒？哪个镜该静、哪个该动？" | 分镜逻辑 + 每镜时长 + 运动类型（静/动/半动） | 每镜是否只讲一件事——一镜塞两个以上知识点=未分镜 |
 | **旁白策略** | "旁白用什么语气？跟画面怎么对齐？原文能直接用吗？" | 语气定义 + 每镜旁白文本 + 与画面的时间对应关系 | 旁白是否可直接录制——需要再改写=不合格 |
 
 ## 三、五轮对话中的门控嵌入点
 
 | 轮次 | 用户指令中应自然包含的门控 | 对应 visual-lab 层 |
 |---|---|---|
 | 第一轮 | 给内容原文、说目的和平台、描述之前的失败症状 | 八变量对齐 → 触发诊断 |
 | 第二轮 | 锁定画幅/风格底线、列出"绝对不能错"的核心事实 | 绝对锁定 |
 | 第三轮 | **要求诊断失败原因**（不是要求改进）、要求按知识点拆镜 | 诊断顺序 + 分镜与节奏 |
 | 第四轮 | 要求每镜包含：画幅、锁死事实、要记住什么、具体构图、光影材质、文字谁做、禁止什么 | 创意母题 + 构图骨架 + 材质光影 + 文字策略 + 禁止项 |
 | 第五轮 | 执行 + 逐镜验收标准 | 绝对锁定 → 事实验收 |
 | 第六轮 | 要求视频：区分静/动镜、锁定运动方向和时间线、写转场方式 | 分镜与节奏（动态） |
 | 第七轮 | 要求旁白=原文不改写、背景音不喧宾夺主、时序对齐 | 旁白策略 |
 
 ## 四、第四轮质量自检（达人看案例时用）
 
 此环节写入案例脚本，但不喂给豆包——它是给**看案例的达人**看的，告诉他"你不需要懂 prompt 工程，你只需要查这几点就能判断豆包写得好不好"。
 
 ```
 （内心旁白，不对豆包说）
 我现在快速看一下豆包写的提示词：
 1. 说清楚这个视频在哪用了吗？→ 教室投影 16:9 ✓
 2. 绝对锁定了几个事实？→ 数一下，3个以上才靠谱
 3. 每镜能一句话说清让人记住什么吗？→ 读一遍，听不懂的就是废的
 4. 禁止项是不是"不要丑"这种废话？→ 是废话就让它重写
 5. 文字策略有没有说"小字后期处理"？→ 没说=默认AI写字，大概率翻车
 6. 镜头运动有没有区分哪些镜该静哪些该动？→ 全推拉=没节奏
 → 6条都过 = 可以执行。没过 = 哪条没过让它改哪条。
 ```
 
 ## 五、常见翻车对照
 
 | 达人容易犯的错 | 表现 | 门控如何拦截 |
 |---|---|---|
 | 只给内容不给约束 | "帮我做个潜水艇视频" | 第一轮要求说清平台/比例/受众 |
 | 优化方向跑偏 | 画面不好了加装饰（水母/珊瑚） | 第三轮先诊断再改——"问题是画面不丰富还是物理事实没锁？" |
 | 不分镜 | 15秒视频一条prompt | 第三轮要求按知识点拆镜 |
 | 禁止项是废话 | "不要丑""要好看" | 第四轮自检第4条——直接判不合格 |
 | 旁白后期再想 | 视频出完了才加旁白 | 第七轮必须在执行前写好旁白文本+时间轴 |
