# 提示词双模式与普通用户共创流程

## 一、模式选择

### 模式A：普通用户共创模式（默认）

适用于没有明确要求“一段提示词直接执行”的绝大多数案例。

目标不是替用户提前设计好全部细节，而是模拟普通人真实使用豆包的过程：先讲困扰，豆包提出办法，双方逐步补全，最后由豆包自己写出详细执行提示词并调用Skill。

用户未指定模式时，必须默认使用模式A，不要反问用户选哪种模式。

### 模式B：专家直出模式（保留）

仅在以下情况使用：

- 用户明确说“直接给我一段完整提示词”“一轮执行”；
- 需要固定输入、固定交互和固定真值的回归测试；
- 需要验证物理、计算、法律条款或数据口径等高风险细节；
- 用户已经给出成熟方案，只需要整理成可执行提示词；
- 用户明确要求两种模式都给。

模式B可以像历史案例一样，由案例设计者直接写出完整解析规则、交互、视觉、边界和验收要求。

## 二、模式A的五轮流程

### 第一轮：只说痛点

用第一人称说一件具体的麻烦，像普通用户聊天，不提图表、信息架构、renderer或复杂功能。

示例：

```text
我收到了四家供应商的报价，每家的表格格式都不一样。有的含税，有的不含税，还有的价格很低但像是漏了东西。我自己对了半天还是怕选错。文件都上传了，你先帮我看看这种情况能怎么处理，先不用急着做成品。
```

### 第二轮：补充真正想解决的问题

说清最终要做什么判断，不指定实现方式。

```text
我真正想知道的不是谁报价最低，而是按我的需求全部补齐后，每家最后要花多少钱。最好还能看出漏项、降配和说不清楚的地方。你觉得做成什么样最合适？
```

### 第三轮：让豆包自己设计

明确用户不懂设计，让豆包决定信息顺序、交互和取舍。

```text
就按你最推荐的办法来。我不太懂图表和页面设计，你继续想具体一点：打开以后先看什么、怎么操作、怎样避免出错。不要堆功能，先说方案，还不用执行。
```

豆包只有在答案会明显改变结果时才追问，最多3个问题。其余内容根据文件和常见场景自行判断。

### 第四轮：让豆包自己写详细提示词

```text
把刚才的方案整理成一份完整提示词，写得详细一些，后面你自己可以直接照着执行。文件怎么读、数据怎么核对、重点展示什么、用户怎么操作、如何回到原文、遇到不确定内容怎么办，都要写清楚。现在先发提示词，不要执行。
```

这一步产出的长提示词来自豆包自己的理解和前几轮讨论，不是案例设计者预先塞给它的答案。

### 第五轮：确认并执行

```text
这份提示词可以。现在严格按照它执行，必须调用【Skill名称】，直接生成完整结果，不再修改需求。
```

Skill名称尽量到这一轮才出现。这样前面的对话更像普通用户，也便于用同一需求公平比较不同Skill。

## 三、同题比较两个Skill

当用户希望比较`doubao-visualization`和`doubao-app-builder`时：

1. 分别新开对话；
2. 上传完全相同的文件；
3. 前四轮逐字保持一致；
4. 对豆包追问使用相同原则回答；
5. 第五轮只替换Skill名称；
6. 同时保留豆包第四轮写出的提示词和最终产物。

不要提前把Visualization限制成“一张图”，也不要强迫App Builder加入数据库、登录和表单。先让Skill自己表现边界。

比较三层差异：

- 豆包如何理解用户痛点；
- 豆包自己写出的详细提示词有什么不同；
- 最终产物更偏一次性理解，还是持续录入和管理。

如果两边仍然很像，这也是有效测试结论，不要为了证明差异而事后改提示词。

## 四、模式B的基本结构

专家直出提示词按风险选择以下模块，不要求机械凑齐：

1. 必须调用的Skill和禁止路由；
2. 文件读取、唯一ID和数据核对规则；
3. 业务目标和核心视觉概念；
4. 首屏信息、主要视图和交互；
5. 原文、条款或数据证据追溯；
6. 视觉系统与响应式要求；
7. 不确定内容、错误和缺失值处理；
8. 输出类型和能力边界；
9. 预期真值、测试步骤和验收表。

模式B仍需使用人话描述业务，不得用长提示词掩盖案例本身不成立。

## 五、案例交付结构

新案例通过后，默认交付：

1. 一句话案例和普通人痛点；
2. 造假背景与附件；
3. 模式A五轮对话脚本；
4. 豆包追问时的统一回答原则；
5. 最终Skill执行句；
6. 独立于用户提示词的预期真值与验收项。

只有命中模式B条件时，再提供案例设计者写好的完整直接执行提示词。历史案例中的长提示词可以保留，视为模式B和回归测试基线，不要求全部重写。

## 六、普通用户语言检查

模式A前四轮必须通过以下检查：

- 是否像一个真实的人在说自己的麻烦；
- 是否先说为什么难，而不是先报Skill名称；
- 是否避免“信息架构、数据编码、renderer、状态机”等专业词；
- 是否把“交互设计”说成“我打开先看到什么、可以点什么”；
- 是否没有提前指定图表、配色和页面布局；
- 是否给豆包留下自己判断和取舍的空间；
- 是否让豆包承担把简单需求变成详细提示词的工作。

如果第一轮就已经像专家验收文档，说明误用了模式B。
