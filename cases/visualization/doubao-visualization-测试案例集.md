## 案例一：科研实验数据交互分析（学术场景）

### 一、案例设计

| 维度 | 内容 |
|------|------|
| **白名单触发** | 条件2：数据趋势/对比/波动/相关性或多指标对比 |
| **目标人群** | 研究生、科研人员、实验室PI——每天面对实验原始数据，需要快速探索变量间关系、发现异常点 |
| **核心痛点** | 传统做法：数据导出→Excel/Origin/GraphPad→手动调参画图→截图贴论文。每次想换一个变量看关系都得重画一张。关键是在截图上看不出交互——不能hover看具体数值、不能框选局部放大、不能点图例筛选数据系列 |
| **visualization的不可替代性** | ECharts交互图表可以让研究者**在对话里直接探索数据**——hover每个散点看x,y值、框选异常区域放大、点击图例只看某个剂量组。这是静态图片做不到的 |
| **一句话传播** | "别人在Excel里手动调参画图，AI直接在聊天里给你生成了交互图表——hover看数值、框选放大、点图例筛选——论文图表的数据探索在对话里就完成了" |
| **可量化利益** | 每次数据分析省1-3小时制图+调参时间；异常数据点不再漏看 |

**叙事弧（30秒）：**
- 0-5s 痛苦：一个研究生对着Excel里三组实验数据，手动做散点图、调坐标轴、标异常点
- 5-12s 展示：她在豆包里说"帮我分析这三组药效数据"。AI在聊天里直接渲染了一组交互ECharts——散点图、剂量反应曲线、三组对比箱线图
- 12-20s 交互展示：她hover到右上角的异常点——"剂量50mg，药效值只有32%，比其他点低了一半"。她框选这个区域放大。她点击图例，关掉安慰剂组只看实验组
- 20-25s 对比：左边是Excel截图贴的静态图（信息密度低、不能交互）；右边是豆包回复里的交互图表（hover、缩放、筛选）
- 25-30s 结尾："下次数据更新，直接输入新数据，图表自动更新"

### 二、测试背景信息（造假用）

**实验场景：某药理学研究**
```

实验组：化合物X（低剂量25mg、中剂量50mg、高剂量100mg）
```

### 四、预期Skill路由

```
用户prompt -> 命中可视化白名单条件2"数据趋势/对比/多指标"+"交互图表"
↓
读取 references/visualization-trigger-design.md
↓
涉及ECharts -> 读取 references/visualization-echarts-interaction.md
↓
数据校验：用户提供的是"实验结果数据"，标注为"用户提供数据，未发表验证"
↓
图表库分层：数据图表 -> 选ECharts（柱状图+误差线+散点叠加=需要多系列）
↓
设计令牌应用：深色/浅色按用户指定
↓
稳定性检查：IIFE、try-catch、DOM检查、CDN降级、无地图
↓
ECharts专项：
  - 误差线用series -> type:'custom' renderItem绘制
  - 散点叠加用第二个series -> type:'scatter'
  - tooltip formatter显示均值/SEM/n
  - 图例点击切换
  - 异常值markPoint标注
↓
输出：文字结论 + html type="renderer"（<=300行，多图表需拆分）
```

### 五、测试关注点

1. **误差线渲染**：ECharts本身不支持原生error bar。Skill需要自己用custom series或scatter+线段实现误差线——这是核心验证点
2. **散点叠加**：散点和柱状图是否能正确叠加在同一个grid内，x轴对齐
3. **异常值标注**：中剂量组32%是否被正确标红/高亮
4. **tooltip信息密度**：hover一个柱子时是否显示均值、SEM、n值的完整信息（而非只显示一个数字）
5. **统计显著性标注**（提示词C）：是否能在两组之间添加*号标注
6. **深色主题科研风格**：提示词C要求的"GraphPad风格"是否被执行

## 案例二：潮汐现象交互演示（教育场景——优先）

### 一、案例设计

| 维度 | 内容 |
|------|------|
| **白名单触发** | 条件3：教学型概念且强依赖动态观察（参数变化->曲线/轨迹/分布/空间关系变化） |
| **目标人群** | 地理老师、科普内容创作者、自然教育从业者——需要向学生/观众解释潮汐的形成机制 |
| **核心痛点** | 课本上是静态图——画了月球、地球、潮汐隆起，学生只能看到"一个瞬间"。但潮汐本质是**动态过程**：月球绕地球转、潮汐力随位置变化、地球自转导致各地潮汐位相不同。静态图无法传达"为什么一天有两次潮汐""为什么每月有大潮小潮" |
| **visualization的不可替代性** | Canvas动画+参数滑块——可以**拖动月球位置看潮汐力实时变化**、**调整月-日夹角看大潮小潮**、**播放动画看潮汐隆起如何跟随月球移动**。这是静态图片和视频都做不到的——视频是线性的，交互是探索式的 |
| **一句话传播** | "地理老师再也不用PPT画箭头解释潮汐了——AI直接生成一个交互演示，学生自己拖动月球、调整参数、看潮汐力实时变化" |
| **可量化利益** | 省去做教学动画的2-5小时；学生从"被动看视频"变成"主动探索"，理解深度完全不同 |

**叙事弧（30秒）：**
- 0-5s 痛苦：地理老师在黑板上画地球+月球，画两个潮汐隆起，画箭头解释引力——学生们一脸"懂了但没完全懂"
- 5-12s 展示：她在豆包里说"帮我做一个潮汐的交互演示"。AI在聊天里直接渲染了一个交互可视化——地球在中间，月球在轨道上，潮汐隆起随月球位置实时变化
- 12-20s 交互展示：她拖动"月-日夹角"滑块——45度（小潮），潮汐隆起比较均匀；0度（大潮），隆起最大。她点"播放"按钮——月球绕地球转，潮汐隆起跟着走，地球自转导致各地经历涨潮落潮
- 20-25s 对比：左边是地理课本上的静态示意图；右边是豆包回复里的交互演示——同一个概念，完全不同的理解效率
- 25-30s 结尾："以后讲任何地理现象，AI帮你生成交互演示"

### 二、测试背景信息（造假用）

**教学场景：高中地理《潮汐现象》**
```
教师：张老师，高中地理教师，教龄8年
课程：高中地理必修一《地球的运动》-> 潮汐现象
学生痛点：不理解为什么一天有两次涨潮落潮、不理解大潮小潮为什么和月相有关
现有教具：课本插图+一个YouTube视频链接（但不能交互）
```

**潮汐物理原理（用于指导skill生成，不是给用户的提示词）：**
```
- 地球自转周期24小时，月球绕地球公转周期约27.3天
- 潮汐力来自月球和太阳的引力梯度
- 朝向月球一侧：引力强->海水隆起（涨潮）
- 背向月球一侧：离心力效应->海水也隆起（涨潮）
- 垂直于月球方向：海水下降（落潮）
- 大潮：月球和太阳在一条线上（新月/满月），潮汐力叠加
- 小潮：月球和太阳成90度（上弦月/下弦月），潮汐力减弱
```

### 三、测试提示词

**提示词A（基础版——直接描述需求）：**
```
帮我做一个潮汐现象的交互演示，给高中生讲课用。

需要展示：
- 地球在中间（不用精确地图，简单的圆形就行）
- 月球在地球外面绕轨道运行
- 地球表面的海水随月球引力形成潮汐隆起（两个隆起，一个朝向月球，一个在对面）
- 能拖动月球位置，看潮汐隆起如何跟随变化
- 有一个"自动播放"按钮让月球绕轨道走
- 显示潮汐力的方向（用箭头表示）

让学生能直观理解为什么一天有两次涨潮落潮。
```

**提示词B（进阶版——带大潮小潮+参数调节）：**
```
做一个潮汐的交互教学演示，有两个调节维度：

基础设定：地球在中间（蓝色圆形），月球绕地球公转，太阳在远处（右边固定位置）

交互1：月球位置滑块（0-360度），拖动滑块时月球在轨道上移动，潮汐隆起实时跟随
交互2：月-日夹角显示（0度=大潮，90度=小潮），数值旁边标注"大潮"或"小潮"
交互3："播放"按钮让月球自动绕转，速度可调（慢/中/快三个挡位）

视觉要求：
- 地球上的海水用浅蓝色椭圆变形表示（朝向月球和反方向两个隆起）
- 潮汐隆起的高度随月球距离变化（月球越近隆起越大）
- 月球引力方向用箭头从地球指向月球
- 显示"月球引力"和"离心力"两个标注
- 浅色背景，课堂投影仪上要能看清
```

**提示词C（地理教研场景——实时参数显示）：**
```
做一个地理课堂用的潮汐交互模型。

核心交互：拖动月球位置（0-360度对应一个月球公转周期的不同时刻）

需要实时显示的内容：
- 月球位置角度
- 潮汐力大小（相对值）
- 大潮/小潮指示（基于月-日夹角）
- 地球上某个固定点（比如A点）当前是涨潮还是落潮
- 一天内A点经历几次涨潮落潮

要求：
- 月亮绕行方向标注
- 关键位置标注：新月（0度）和满月（180度）位置特别标出"大潮"
- "自动演示"按钮，速度可调
- 投影仪视觉效果：色彩对比度要高，字体不小于14px
```

### 四、预期Skill路由

```
用户prompt -> 命中白名单条件3"教学型概念+动态观察"
↓
读取 references/visualization-trigger-design.md
↓
涉及动画循环+滑块交互 -> 读取 references/visualization-echarts-interaction.md 第十节
↓
交互形态选择：
  - 月球位置拖动 -> 形态1连续滑块 + 形态2拖拽锚点
  - 自动播放 -> 形态4动画循环（requestAnimationFrame + 暂停/继续）
↓
设计令牌应用：字体不小于投影仪14px、高对比度配色
↓
稳定性检查 + 动画专项：
  - 必须可暂停可重置
  - requestAnimationFrame有停止条件
  - 滑块拖动时实时更新（不卡顿）
  - 不使用100vh/固定大高度
↓
输出：文字解释 + html type="renderer"
```

### 五、测试关注点

1. **动画流畅性**：月球绕轨道动画是否平滑（requestAnimationFrame），速度调节是否即时生效
2. **潮汐隆起的物理正确性**：两个隆起是否分别朝向月球和反方向，隆起幅度是否随月-地距离变化
3. **大潮/小潮判断**：月-日夹角计算是否正确（0度/180度=大潮，90度/270度=小潮）
4. **暂停/继续/重置**：动画控件是否都有
5. **投影仪适配**：画布大小是否足够大（至少800x600以上）、颜色对比度是否够高
6. **移动端兼容**：学生在手机上拖滑块是否顺畅（触控点>=44x44px）

## 案例三：化学反应配平步骤演示（教育场景）

### 一、案例设计

| 维度 | 内容 |
|------|------|
| **白名单触发** | 条件4：步骤状态机（算法逐步执行、状态迁移、流程推进——用户目标是"看过程"） |
| **目标人群** | 化学老师、中学理科教师、化学家教 |
| **核心痛点** | 化学配平是基础技能，但"最小公倍数法"配平的中间步骤学生只能在草稿纸上自己推理。PPT只能展示"配平前"和"配平后"两页，中间试错过程看不到。老师想给学生演示"怎么一步一步找到系数"——这是步骤状态机的教科书级应用场景 |
| **visualization的不可替代性** | 按钮推进状态——每点一次"下一步"，化学反应式更新一个系数、显示当前原子数对比、标注接下来该配平哪个元素。学生可以反复回退、重看、自己尝试 |
| **一句话传播** | "化学老师解释配平再也不靠黑板画箭头了——AI生成一个交互演示，学生自己点'下一步'看每次系数变化，能回退、能重来、每一步都显示原子数对比" |
| **可量化利益** | 备课时省去做配平动画/找视频的时间；学生自主练习效率提升 |

**叙事弧（30秒）：**
- 0-5s 痛苦：化学老师在黑板上写Fe2O3 + CO -> Fe + CO2，一步一步试系数——擦了写、写了擦，手里还拿着参考书确认步骤
- 5-12s 展示：她在豆包里说"帮我做一个铁矿石还原反应配平的交互演示，要能一步一步展示最小公倍数法"。AI在聊天里渲染了一个交互面板
- 12-20s 交互展示：学生点"下一步"——第一步锁Fe，看原子数、第二步配O，显示最小公倍数计算、第三步检验——每一步都能看到当前式子、左侧原子计数
- 20-25s 对比：左边是课本上的静态配平过程（学生看不懂跳跃）；右边是交互演示（每一步都有原子数对比、高亮显示当前操作的元素）
- 25-30s 结尾

### 二、测试背景信息

**教学场景：高中化学《铁矿石还原》**
```
反应：Fe2O3 + CO -> Fe + CO2
配平方法：最小公倍数法（氧化还原法也可）
关键步骤：
1. 找最复杂分子：Fe2O3（含2个Fe、3个O）
2. 配Fe：左边2个Fe -> 右边需要2个Fe（系数2）
3. 配O：左边3个O（来自Fe2O3）+ x个O（来自xCO）= 右边2x个O（来自xCO2）
   解：3 + x = 2x -> x = 3
4. 最终：Fe2O3 + 3CO -> 2Fe + 3CO2
5. 检验：Fe:2=2 ok, C:3=3 ok, O:3+3=6=6 ok
```

### 三、测试提示词

**提示词A：**
```
帮我做一个化学反应配平的交互演示，用最小公倍数法。

反应：Fe2O3 + CO -> Fe + CO2

要能一步一步展示配平过程：
第1步：锁定Fe2O3，显示当前原子数（左边Fe=2, O=3, C=1 / 右边Fe=1, O=2, C=1）
第2步：配平Fe，右边Fe系数改为2
第3步：配平O和C，左边CO系数改为3，右边CO2系数改为3
第4步：检验，所有原子数相等

每一步都要显示：
- 当前的化学方程式（系数变化高亮）
- 左边 vs 右边原子数对比表
- 当前正在操作哪个元素

要有"下一步"和"上一步"按钮，让学生逐步理解配平逻辑。
```

**提示词B（进阶——含氧化还原电子转移标注）：**
```
做一个铁矿石还原反应的交互配平演示。

反应：Fe2O3 + CO -> Fe + CO2

除了逐步配平系数（最小公倍数法），还要展示氧化还原的电子转移：
- Fe从+3价降到0价（每个Fe得3个电子，共6个电子）
- C从+2价升到+4价（每个C失2个电子，共6个电子）

每一步：
1. 显示当前化学方程式
2. 显示左边vs右边原子数对比
3. 标注化合价变化（被还原的元素=Fe标蓝、被氧化的元素=C标红）
4. 显示电子转移数

交互：下一步/上一步/重置
```

```
↓
```

### 五、测试关注点

1. **状态推进正确性**：每一步系数是否正确更新、原子数对比是否准确
2. **高亮标记**：当前操作的元素是否被高亮、系数变化是否明显
3. **上一步/回退**：是否能回到上一步且状态完整恢复
4. **重置按钮**：是否回到初始状态

## 测试执行建议

### 测试优先级

建议按 **案例三 -> 案例一 -> 案例二 -> 案例四 -> 案例五** 的顺序：

1. **案例三（化学配平）**——最简单、纯逻辑、不依赖数据，先验证基础步骤状态机能力
2. **案例一（药效数据）**——测试ECharts核心能力（误差线、散点叠加、异常标注、多系列联动）
3. **案例二（潮汐演示）**——最复杂、测试Canvas动画循环+交互控件+物理正确性，是客户的优先需求
4. **案例四（甘特图）**——打工场景首次测试
5. **案例五（问卷看板）**——多图表组合+雷达图+环形图

### 特别提醒

1. **不要预生成图表PNG**：必须让skill内部渲染ECharts/SVG/Canvas
2. **动画必须可暂停可重置**：案例二的动画循环和案例三的步骤推进都必须有暂停/恢复/重置控件
3. **不要用100vh**：renderer容器高度由内容自撑
4. **数据真实性声明**：所有案例数据标注为"用户提供，未独立验证"
5. **单HTML块<=300行**：多图表场景必须拆分
6. **地图绝对禁止**：即使案例二涉及地理概念，也不能渲染地图
