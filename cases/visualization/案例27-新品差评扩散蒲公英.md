# 豆包 Visualization 测试案例：新品差评扩散蒲公英

> 状态：造假Excel与测试提示词已完成，待豆包实测
> Skill路由：`doubao-visualization` -> 对话内可交互HTML renderer
> 输入方式：上传1份多平台用户反馈Excel，一次生成
> 推荐端侧：PC端
> 边界：一次性反馈分析，不做舆情监控系统、客服工单库或长期数据后台

## 一、案例一句话

新品上线后，运营收到来自天猫、京东、抖音、小红书和品牌小程序的180条差评。把Excel交给豆包，直接生成一株“差评扩散蒲公英”：根茎追到批次、仓库和渠道，花盘聚合问题类型，每颗绒球是一条真实反馈，飞出去的种子则是正在放大影响的高互动内容。

传播口径：

> 新品上线两周有180条差评，口味问题看着最热闹，但真正该先处理的是包装漏液：48条里36条指向同一批次，退货率超过七成。以前要筛半天表格，现在把Excel扔给豆包，一株蒲公英就把问题从哪儿长出来、又往哪儿扩散画清楚了。

## 二、为什么这个需求成立

售后和商品运营面对的难点不是“数一共有多少条差评”，而是同时回答四个问题：

1. 哪类问题数量最多？
2. 哪类问题真正造成退货、投诉和退款？
3. 问题是否集中到某个批次、仓库或渠道？
4. 哪些少量高互动反馈正在把问题带出原有用户圈层？

普通柱状图只能回答第一个问题；词云无法连接原话、批次与商业后果；传统网络图又很容易变成一团乱线。蒲公英结构适合把“根因—问题簇—单条反馈—向外扩散”放进同一张稳定图中。

## 三、为什么属于Visualization

- 用户上传一批现成反馈，目的是一次性理解当前问题结构；
- 交互用于筛选、追溯、比较和播放时间演化；
- 不需要继续录入评价、分配工单、保存处理状态或自动抓取平台；
- 因此应调用`doubao-visualization`，不应调用`doubao-app-builder`。

如果后续需要每日自动抓取、持续更新、客服认领和处理闭环，应另做应用。本案例不要越界。

## 四、PC端适配判断

- PC-1：命中。运营通常在电脑上处理平台导出表和售后明细。
- PC-2：命中。180条反馈、30个字段及多层关系不适合手机首屏承载。
- PC-3：部分命中。实务中常需在平台后台与分析结果之间切换，但本测试只上传Excel。

推荐PC端为主，移动端用于查看聚焦后的单个问题簇。

## 五、造假文件

上传文件：

`澄野气泡水-多平台用户反馈明细-虚构测试.xlsx`

工作表：

- `使用说明`：数据范围、使用方式与虚构声明；
- `概览校验`：公式驱动的总量、退货、投诉、退款和问题分布；
- `用户反馈明细`：180条反馈原文、平台、地区、批次、履约仓、严重度和处理结果；
- `根因映射`：疑似根因、责任团队、数据证据与建议动作；
- `可视化口径`：每个字段应如何映射到蒲公英结构。

所有品牌、订单、用户、评价、金额和批次均为虚构。文件中的根因属于测试设定，不是实际质量结论。

## 七、直接测试提示词

上传Excel后，直接发送以下整段：

```text
请必须调用doubao-visualization，读取我上传的《澄野气泡水-多平台用户反馈明细-虚构测试.xlsx》，生成一份对话内可交互的“新品差评扩散蒲公英”。不要调用app-builder，不要做客服工单系统、舆情监控后台、数据录入页或静态海报。

这是一份虚构测试数据。你的任务是帮助售后和商品运营理解问题结构，不要替用户直接宣布产品召回、供应商责任或确定因果。

【数据解析规则】
1. 以“用户反馈明细”为原始事实表；“概览校验”用于核对总量和聚合结果；“根因映射”提供测试设定的疑似根因；“可视化口径”规定字段与图形的对应关系。
2. 每个反馈ID只能生成一个原始反馈节点，不能因为同时属于平台、地区和批次而重复画多次。
3. 严格区分反馈数量、节点权重、互动量、退货数量、退款金额和投诉升级。默认主视图用“节点权重”，但必须允许切换口径。
4. 节点权重读取Excel公式结果，不自行重算另一套黑盒分数；显示权重时解释其由严重度、互动量、退货和投诉升级共同构成。
5. “根因映射”中的内容统一标为“疑似根因/数据关联”，不能写成已证实因果。证据置信度高用实线、中用短虚线、低用点线。
6. 所有结论都必须能追溯到反馈ID、原始反馈文字和Excel字段；不确定内容标“待确认”。

【核心视觉概念】
把这批新品差评翻译成一株“从供应链长出、被平台风吹散的蒲公英”：
- 根部是批次、履约仓和下单渠道，表示问题可能从哪里进入；
- 花盘按一级问题分成7个稳定扇区，扇区面积按反馈数量；
- 每颗绒球是一条反馈，大小按当前选择的分析口径；
- 互动量高或达人反馈形成10颗飞散种子，离开花盘但保留到原问题簇的细线；
- 根茎旁的压力刻度显示退货数量或退款金额，表示商业后果。

不要使用普通力导向网络图。主图布局必须稳定、可复现，筛选和口径切换不能让整株蒲公英重新随机乱跳。

【首屏信息】
首屏直接给出一句基于数据的发现：
“包装破损/漏液有48条，其中36条落在B26-0712批次，退货率70.8%；它比数量第二的口味后苦更值得优先排查。”

同时以编辑式数字摘要显示反馈总数、退货申请、投诉升级、飞散高影响节点和退款金额，但不要做五张等宽圆角KPI卡。

【主视图编码】
1. 花盘中心显示产品名与“上线后14天”，外圈按反馈日期由内向外排列，越晚出现的反馈越靠外。
2. 一级问题决定花盘扇区；反馈ID决定唯一节点；二级问题决定节点周围的细短绒毛方向。
3. 默认节点大小使用“节点权重”；可切换为反馈数量等权、互动量、严重度、退货压力。
4. 节点颜色表示处理结果：仅公开反馈、客服处理中、退款/退货、投诉升级；颜色之外再用图形边框或符号编码，不能只靠颜色。
5. 平台使用节点外缘纹理或小型字母标记区分，避免五种平台颜色与处理结果颜色冲突。
6. 批次、履约仓和下单渠道作为根系节点。点击根系时，高亮所有相关反馈并显示“涉及反馈数/总数”和证据置信度。
7. 飞散种子只代表高互动或达人反馈，不能解释为内容更真实或问题更严重。

【交互】
1. 提供“反馈数量 / 节点权重 / 互动量 / 严重度 / 退货压力”分段切换。切换只改变编码和排序提示，不改变问题扇区的位置。
2. 提供平台、地区、一级问题、处理结果、批次、履约仓和证据置信度筛选。
3. 提供“只看飞散高影响”“只看退货/投诉”“只看高置信关联”开关。
4. 提供7月1日至7月14日的时间播放。按天播放时，反馈节点依次长入花盘，高影响节点在其发布日期飞出；暂停后可拖动日期。
5. 悬停节点显示反馈ID、日期、平台、地区、原文短句、严重度、互动量和处理结果。
6. 点击节点锁定并展开侧栏，显示完整反馈原文、批次、仓库、渠道、退款、投诉、根因证据、置信度和关联路径。
7. 点击问题扇区进入聚焦模式，只保留该问题的根系、节点和飞散种子；提供返回全景按钮。
8. 允许任选五个问题中的两个做对比，显示数量、平均评分、退货率、投诉升级和平均权重；不要用单一总分宣布优先级。
9. 增加“数量最多 / 退货压力最大 / 扩散最远”三个故事书签，点击后自动聚焦对应视角并展示解释。

【信息层级】
主视图占画面约70%，让蒲公英成为唯一视觉中心。左上是标题和一句洞察；右侧是可收起的原话与根因追溯栏；底部是时间尺、图例和口径说明。筛选控件集中成一条窄工具栏，不要散落成卡片阵列。

【视觉系统】
背景使用偏冷的雾白纸张色，花盘以薄荷绿、灰绿和少量冷青构成；退款/投诉使用克制的朱红，高影响飞散种子用暖琥珀。根系使用深墨绿，文字使用近黑。整体像成熟的数据新闻图解，细线、注释、比例尺和原话引线承担信息组织。

节点可以有轻微透明度和层次，但禁止玻璃拟态、霓虹外发光、深蓝/紫色渐变、漂浮光斑、无意义粒子动画和装饰性3D。圆角控制在0-6px。不要为了“蒲公英”生成写实植物插画，所有形态必须由真实数据编码产生。

【响应式】
桌面端显示完整蒲公英、根系和侧栏；窄屏默认进入问题聚焦模式，只显示一个扇区，顶部提供问题切换，侧栏改为底部抽屉。任何宽度下不能出现节点标签互相遮挡、原文溢出、筛选控件换行压住主图或动画导致布局跳动。

【输出要求】
直接输出可交互HTML renderer。先核对Excel公式与原始明细，再渲染。不要只给文字结论、柱状图Dashboard、词云、静态图片或设计建议。
```

## 九、测试流程

### 第一步：路由与数据核对

上传Excel，发送完整提示词。

观察：

- 是否调用`doubao-visualization`；
- 是否识别180条唯一反馈、7类问题和10条飞散高影响反馈；
- 是否核对退货68、投诉33、退款3,192元；
- 是否没有做成应用后台或普通柱状图Dashboard。

### 第二步：花盘与原话追溯

点击“包装破损/漏液”扇区，再选择任意反馈节点。

观察：

- 是否只出现48个唯一节点；
- 是否能看到完整原话、反馈ID、批次、仓库与处理结果；
- 是否能高亮36条B26-0712关联记录；
- 是否把关联写成疑似，而不是确定因果。

### 第三步：口径切换

依次切换“反馈数量”“节点权重”“退货压力”。

观察：

- 花盘扇区位置是否保持稳定；
- 数量模式下包装48、口味37；
- 退货压力模式下包装问题是否明显加强；
- 页面是否解释了每种口径，而不是只让圆点变大。

### 第四步：飞散种子

打开“只看飞散高影响”，再按平台筛选抖音和小红书。

观察：

- 是否只剩10条高互动或达人反馈；
- 每条种子是否保留到原问题簇的连线；
- 是否能看到互动量、达人标记与原文；
- 是否避免把高互动自动写成高严重度或高真实性。

### 第五步：时间播放

从7月1日播放到7月14日，中途暂停并拖动日期。

观察：

- 节点是否按真实反馈日期进入；
- 飞散种子是否在自身日期出现；
- 暂停和拖动后视图是否同步；
- 动画是否只帮助理解传播顺序，没有无意义飘动。

### 第六步：双问题对比

对比“包装破损/漏液”和“口味后段发苦”。

观察：

- 是否显示48对37、1.79对3.43、70.8%对27.0%、11对5；
- 是否能得出“包装优先排查，但口味仍需研发验证”的辅助结论；
- 是否没有把平均权重包装成不可解释的总评分。

## 十一、常见翻车

1. 只画一个漂亮蒲公英，节点大小和颜色没有数据含义。
2. 用力导向网络图随机摆放，筛选一次整张图重新跳动。
3. 同一反馈因属于多个维度被重复画成多个节点。
4. 把互动量直接当成严重度，把飞散距离直接当成事实可信度。
5. 把根因映射里的“疑似”删掉，直接宣布批次或仓库负责。
6. 只展示AI摘要，点击节点看不到原始反馈。
7. 颜色同时编码平台、问题和处理结果，导致图例失控。
8. 时间播放只是装饰动画，没有按反馈日期出现。
9. 做成深色霓虹舆情大屏，真正需要看的批次和原话反而藏起来。
10. 越界做成带工单分派、账号和数据库的客服系统。

## 十二、30秒演示脚本

> 这是新品上线两周的180条差评。正常做法是先按问题分类，再筛批次、仓库和退货，表格翻半天。现在我把Excel丢给豆包，它生成了一株差评蒲公英。每颗绒球就是一条原话，飞出去的是高互动反馈，下面的根能追到批次和仓库。点包装漏液，48条里有36条指向B26-0712，退货率70.8%；再点口味后苦，虽然数量第二，但退货压力明显小。它没有替我宣布是谁的责任，但已经把最该先查的链路摆在眼前了。
