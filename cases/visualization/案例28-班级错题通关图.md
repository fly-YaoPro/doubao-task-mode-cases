# 豆包 Visualization 测试案例：班级错题通关图

> 状态：造假试卷、答题明细与测试提示词已完成，待豆包实测
> Skill 路由：`doubao-visualization` -> 对话内可交互 HTML renderer
> 输入方式：同时上传 1 份 Word 试卷与 1 份 Excel 答题明细，一次生成
> 推荐端侧：PC 端
> 边界：用于一次考试的教学复盘，不做教务系统，不长期保存学生档案，不给学生贴能力标签

## 一、案例一句话

把一份试卷和全班答题明细交给豆包，它不再只画成绩排名，而是把每道题拆成“读题、概念识别、建立关系、列式、计算、检查”六个关口，直接显示全班究竟在哪一步开始掉队。

传播口径：

> 一道题错了，可能是看不懂，也可能只是最后算错。只看分数，这两种学生完全分不出来。现在把试卷和答题表一起丢给豆包，它会把18道题变成一张解题通关图：第16题有20个人没完全做对，但真正需要重讲的不是计算，而是前面的“建立关系”。点进去还能看到对应学生、作答摘要和评分步骤。

## 二、为什么属于 Visualization

这次任务是对一次考试材料的一次性解释与探索：

- 上传现成试卷和成绩明细后，直接得到可交互结果；
- 交互用于理解题目、错因、知识点和学生之间的关系；
- 不需要账号、数据库、持续录入、作业发布、家校通知或长期学情档案；
- 因此应调用 `doubao-visualization`，不应调用 `doubao-app-builder`。

如果后续需要持续导入每次考试、保存学生历史轨迹、分配补练任务或多人协同，才应另做教学应用。本案例不要越界。

## 三、造假背景

- 学科：八年级数学
- 主题：一次函数阶段检测
- 班级：八年级2班（虚构）
- 匿名学生：36人，编号S01-S36
- 题目：18道，满分100分
- 答题明细：648条，即36人 × 18题
- 检测日期：2026年7月14日
- 班级平均分：72.14
- 及格人数：31人，及格率86.1%
- 最高分：95，最低分：41

所有学校、班级、教师、学生与成绩均为虚构测试数据，不对应任何真实个人。

## 四、造假文件

同时上传：

1. `八年级数学-一次函数阶段检测试卷与评分细则-虚构测试.docx`
2. `八年级2班-一次函数阶段检测-答题明细-虚构测试.xlsx`

Word 包含：

- 18道完整题目；
- 客观题答案；
- 6道解答题的分步评分细则；
- 六类首个错误步骤的标注口径；
- E00-E06错因代码与教师复核动作。

Excel 包含：

- `使用说明`：材料性质和数据口径；
- `题目与步骤`：题号、知识点、难度、满分、解题路径、参考答案；
- `答题明细`：648条逐题作答记录；
- `班级诊断`：题目正确率、主要错误阶段等校验汇总；
- `学生总分`：36名匿名学生总分。

3. [Observable: End-to-end visual data analysis](https://observablehq.com/blog/visual-data-analysis)，2025年5月：借鉴把可视化嵌入分析过程、同时暴露数据口径和潜在误判的方法。
4. [Duolingo Home Screen Redesign](https://blog.duolingo.com/new-duolingo-home-screen-design/)，2022年经典参考：只借鉴清晰的线性学习路径与关卡节点，不使用其品牌角色、插画或视觉资产。该参考超过12个月，但“路径即学习顺序”的机制仍与本案例直接匹配。

## 六、直接测试提示词

上传 Word 和 Excel 后，发送以下整段：

```text

这些是全量虚构测试材料。你的任务是帮助老师理解这一次考试中学生究竟卡在哪个解题步骤，不是给学生贴固定能力标签，也不要把机器错因标签写成未经教师复核的事实。

【数据核对】
1. 先读取Word中的题目、参考答案、分步评分细则和错因标注口径，再读取Excel。
2. Excel应识别36名匿名学生、18道题和648条唯一逐题记录；题目满分合计应为100分。
3. 以Excel“答题明细”为逐题事实来源，以Word评分细则解释每道题的正确解题路径。若两个文件冲突，必须标出冲突，不要静默覆盖。
4. “首个错误步骤”只表示这一次作答中最早出现、并足以影响后续结果的阻塞点。页面统一标注“机器整理，待教师复核”。
5. 不要推断匿名编号背后的真实身份、性别、家庭、智力或长期能力。

【视觉研究】
参考IXL Analytics 2026版中“共同困难→具体错题→学生明细”的下钻逻辑；参考Observable 2025年的“先聚合、交互后展开细节”和可见的数据口径；参考Duolingo学习路径的线性关卡机制。只借鉴这些方法，不得照搬任何品牌界面、插画、图标或配色。

【核心视觉概念】
把一张试卷转译成一张“解题通关路线图”：横向固定为读题、概念识别、建立关系、列式、计算、检查六道关口；每道题是一条细轨道。学生第一次出错的位置形成路障，路障越大，卡在这里的人越多。通过某一步的轨道保持清晰，发生阻塞后的后续轨道降低透明度，让老师一眼看出问题出在理解、建模、表达还是运算。

不要把它做成红绿成绩热力表。主视图必须让“解题顺序”和“首个阻塞点”同时可见。

【信息层级】
1. 首屏左上显示标题“八年级2班｜一次函数错题通关图”和一句基于数据的核心洞察，优先指出第16题的建立关系问题。
2. 标题下只放一条紧凑摘要：36名学生、18道题、班级平均72.14、及格率86.1%。不要拆成四张等宽KPI卡。
3. 页面主体约占70%，显示18条题目轨道与六个步骤关口。
4. 右侧为可收起的“批改夹页”，用于显示选中题目的题干、参考答案、评分步骤、错因人数、匿名学生和作答摘要。
5. 底部放图例、数据口径和“机器整理，待教师复核”说明。

【班级总览】
1. 18条题目轨道默认按原题号排列，同时支持按正确率、错误人数、平均失分和难度排序。
2. 每条轨道左侧显示题号、知识点、题型和满分；右侧显示正确率和错误人数。
3. 关口上的路障面积编码“首个错误步骤人数”，不能把得分、难度和人数混在同一个大小里。
4. 路障颜色按阶段固定：读题/概念使用深蓝，建立关系使用琥珀，列式使用青绿，计算使用朱红，检查使用灰紫；同时用形状或纹理辅助区分，不能只靠颜色。
5. 提供“仅看需要教学干预”开关：默认规则为同题同阶段达到8人，或占全班20%以上。规则必须在页面上可见，允许关闭，不要把它包装成教育学定论。

【三级下钻】
1. 班级层：查看18道题在哪一步集中掉队。
2. 题目层：点击任意轨道或路障，右侧显示完整题干、解题步骤、每一步卡住人数、正确率、平均得分和匿名学生列表。
3. 学生层：点击匿名学生编号，切换为该学生的18题个人路径，显示每题得分和首个错误步骤；不要显示全班名次，也不要使用“差生”“薄弱学生”等标签。
4. 点击右侧评分步骤时，在主图中高亮所有卡在该步骤的题目与学生，帮助老师判断是全班共性问题还是少数人的局部问题。

【关键交互】
1. 筛选：选择题/填空题/解答题、知识点、难度1-5、错误阶段、小组。
2. 排序：原题号、正确率从低到高、错误人数从高到低、平均失分从高到低。
3. 对比：最多选择两道题，比较题干、正确率、平均得分和六阶段阻塞人数。轨道位置保持稳定，不要点击后全图乱跳。
4. 搜索：输入S01-S36匿名编号，快速查看个人路径。
5. 教学分组：点击某个路障后，可临时显示“建议复核名单”；只有达到规则时才提示“适合小组复盘/全班重讲”，并明确这只是基于本次数据的辅助建议，不保存、不自动派发任务。
6. 提供三个故事书签：
   - “全班最难”：聚焦第16题相遇问题；
   - “会做但算错”：聚焦第3题斜率计算；
   - “基础概念误区”：聚焦第1题象限判断。
   点击书签后自动缩放到对应题目并显示一句数据解释，但用户仍可继续自由探索。

【视觉系统】
整体像一张现代数学教研室使用的“透明批改底稿”，不是儿童闯关游戏，也不是企业数据大屏。

背景使用接近白色的试卷纸色#FAF8F2，主文字为墨蓝#17324D，轨道与辅助线为浅灰蓝#CFD8DE。建立关系的重点色为琥珀#D98C2B，计算错误为克制的朱红#B8493F，正确通过使用深绿#3F7C68。颜色总量克制，避免彩虹式六色并排。

标题和正文使用清晰的中文无衬线字体，题号、分数和人数使用等宽数字。结合细网格、批注引线、步骤编号、评分铅笔线和少量圈改痕迹组织信息，但这些元素必须服务于数据定位，不能做成装饰贴纸。

圆角控制在0-6px。禁止深蓝/紫色渐变大屏、玻璃拟态、霓虹发光、漂浮光斑、卡片套卡片、四张KPI卡、雷达图、词云、普通柱状图拼盘、无意义3D和游戏角色插画。

【响应式】

【输出要求】
直接输出可交互HTML renderer。先完成数据核对，再渲染页面。不要只给文字总结、改进建议、Excel表格、成绩排名或静态图片。
```

## 八、测试流程

### 第一步：路由与核数

同时上传Word和Excel，发送完整提示词。

观察：

- 是否调用`doubao-visualization`；
- 是否识别36名学生、18道题、648条记录和100分满分；
- 是否重新计算总体数据；
- 是否没有做成教务后台、成绩排行榜或静态海报。

### 第二步：班级通关图

观察18条轨道和六个固定关口。

观察：

- 六个步骤顺序是否固定；
- 路障面积是否表示人数；
- 第16题“建立关系”是否形成最明显的阻塞点之一；
- 是否能在3秒内看出班级主要卡在建模，而不是被KPI卡和装饰分散注意力。

### 第三步：第16题下钻

点击第16题“建立关系”路障。

观察：

- 是否显示20人未完全答对、10人首错在建立关系；
- 是否显示小周晚出发1小时、统一时间变量和两条路程函数的评分步骤；
- 是否能看到对应匿名学生与作答摘要；
- 是否标注“机器整理，待教师复核”。

### 第四步：对比两种错误

对比第16题与第3题。

观察：

- 第16题是否突出建立关系；
- 第3题是否突出9人的计算错误；
- 页面是否给出不同复核方向，而不是把两题都写成“基础薄弱”；
- 对比后轨道位置是否稳定。

### 第五步：学生路径

任选一个匿名编号，查看18题个人路径，再返回班级总览。

观察：

- 是否只显示匿名编号，不推断真实身份；
- 是否能看到该生每题得分与首个错误步骤；
- 是否没有生成全班名次或人格化标签；
- 返回后筛选和视图状态是否合理保留。

### 第六步：响应式

检查桌面端和窄屏。

观察：

- 桌面端是否保留完整18题主视图；
- 窄屏是否主动降级为六关总览与单题路径；
- 是否无文字遮挡、标签截断、控件溢出和图形重叠。

## 十、常见翻车

1. 只做题目正确率热力表，看不出错误发生在哪一步。
2. 把六类错因做成六种鲜艳颜色，页面像儿童游戏关卡。
3. 把首个错误步骤直接写成学生的固定能力问题。
4. 只读Excel汇总，不读取Word里的题干和评分细则。
5. 把第16题和第3题都归结为“数学基础差”，没有区分建模与计算。
6. 页面塞满平均分、最高分、分数段等成绩指标，真正的解题路径反而变小。
7. 点击题目后整个图重新排序或跳动，用户失去位置感。
8. 越界做成带学生录入、作业派发和长期档案的教务应用。

## 十一、30秒演示脚本

> 一张数学卷子错了，老师最想知道的其实不是谁分数低，而是大家到底卡在哪一步。我把试卷和全班答题表一起丢给豆包，它把18道题拆成了六个关口：读题、概念、建立关系、列式、计算、检查。第16题有20个人没完全做对，其中10个人在“建立关系”这一步就掉队了；但第3题不一样，9个人只是算错。点开第16题，能直接看到题干、评分步骤和对应匿名学生。这样老师知道下一节课该重讲建模，还是只补一组运算练习。
