# 12个案例 × Skill路由可行性逐案审查

> 审查依据：doubao-creative-video/SKILL.md + doubao-creative-design/SKILL.md + 各ref文件
> 审查结论：6案可行，2案需修正，4案有阻塞风险需重新设计


## Skill能力速查（来自实测文档）

### doubao-creative-video 能做什么
- text_to_video / image_to_video：从文字描述或参考图**生成新视频**（基于Seedance模型）
- 三条路线：UGC口播 / Marketing产品广告 / Corporate企业宣传
- **不能做**：对已有视频做剪辑、裁剪片段、提取高光、替换字幕/配音

### doubao-creative-design 能做什么
- image_gen / image_edit：文生图或基于参考图做I2I编辑（去背景、换场景、加文字、调色）
- 六条路由：ecommerce / media-creative / infographic / brand-logo / brand-IP / brand-packaging
- **澄清门控**：最多1轮澄清（ecommerce、media、infographic各有一套澄清规则）
- 关键能力：产品一致性锁（核心资产锁定+edit延展）、平台视觉指纹内置

### 任务模式能叠加什么
- 定时触发：到点自动跑
- 上下文持久化：记住品牌色/产品参数/排版偏好
- 跨工具编排：浏览器+Skill+飞书推送串联

### 关键约束
- Skill的澄清门控在任务模式下是否可跳过？——这是所有批量自动化案例的共性阻塞点
- video skill不能编辑已有视频——案例七、九的核心需求超出了skill能力边界
- image_edit依赖CDN URL——批量自动化中是否能稳定传参？

### 案例一：产品图自动处理流水线

**能力动作**：每周自动→读取文件夹→对每张原图做I2I去背景+换场景→命名归档

**预判路由**：doubao-creative-design → ecommerce-design.md
- 白底主图 → 淘宝主图首图（1:1白底+产品居中）
- 场景图 → 场景带入图（真实使用环境+融入环境的商品）
- 细节特写 → 材质细节屏（局部微距特写）

**路由匹配度**：✅ 高度匹配

**阻塞风险**：
- ⚠️ ecommerce-design有澄清门控（1轮），且在Step 1要求判断"四项关键信息是否齐全"，缺失时会触发追问。批量自动化场景需要跳过澄清直接执行——需验证任务模式是否支持"在创建任务时一次性提供所有参数→后续每次执行不再追问"
- ⚠️ image_edit要求传入核心资产CDN URL，批量处理时每张原图需要先上传获取CDN URL再传edit——如果原图在本地文件夹，能否自动完成"上传→获取URL→edit"的闭环？
- ✅ 产品一致性锁是本场景的核心武器

**判决**：✅ 核心能力可行，需验证两个阻塞点


### 案例二：月度营销视频自动排期

**能力动作**：25号自动→识别下月节日→出脚本→28号出预览视频→节日当天发正式

**预判路由**：doubao-creative-video → Marketing路线 → product-marketing-ad-video-no-storyboard-ref.md
- 唐点的品牌视频定位匹配Marketing路线的"产品营销视频、品牌短片"
- 但Marketing路线需要"产品图+产品核心信息"作为输入——唐点的产品图是否在任务上下文中持续可用？

**路由匹配度**：✅ 匹配
- 但需要注意：Marketing路线的确认门控（主MD要求输出确认摘要→等待用户确认→才调用视频工具）。任务模式下能否在创建时就完成参数确认，后续自动执行？

**阻塞风险**：
- ⚠️⚠️ video skill的核心门控："必须先输出一次参数摘要并等待确认"——这个在任务模式定时触发时会卡住。如果25号自动触发时还要等人确认，就不算"自动化"
- ⚠️ 产品图/品牌资产的门控：Marketing路线要求产品上传或明确brief，任务模式下这些内容需要持久化存储

**判决**：✅ 能力匹配，但"确认门控"是阻塞点需验证


### 案例三：每日客服FAQ自动生成海报

**能力动作**：每天早8:30→读客服聊天记录→统计Top3问题→为每个问题出FAQ海报→推群

**预判路由**：doubao-creative-design
- FAQ海报的产物性质介于两个路由之间：
  - media-creative.md：小红书/朋友圈风格配图
  - graphic-infographic.md：结构化信息图
- 按SKILL.md路由冲突规则："小红书风格FAQ卡片"优先走media-creative

**路由匹配度**：⚠️ 需明确指定

**建议修正**：在提示词中明确写"生成一张小红书风格的快问快答海报"→锁定media-creative路由，避免路由歧义

**阻塞风险**：
- ⚠️ "读取客服聊天记录"这一步不是design skill的能力——需要任务模式先通过浏览器/飞书API读取聊天数据，再把Top3问题喂给design skill
- ⚠️ "已有海报不复用"的判断逻辑——需要任务模式维护一个"已生成FAQ海报库"（上下文持久化）

**判决**：✅ 设计出图能力可行，但"读聊天记录"依赖任务模式的跨工具编排能力

### 案例六：新员工入职物料

### 案例七：服装直播高光切片自动生成 🔴🔴🔴

**能力动作**：每天下播后→读3小时回放→识别高光片段→生成15秒切片→叠加价格/尺码标签

**预判路由**：如果调用doubao-creative-video

**路由匹配度**：🔴 **不匹配——这是致命问题**

原因逐条看：

1. doubao-creative-video的能力是"从brief/文字/参考图**生成新视频**"，底层工具是text_to_video和image_to_video（基于Seedance模型）。它没有"视频理解""视频时间轴定位""视频片段裁剪"的能力。

2. video skill没有任何ref或工具涉及：
   - 对已有长视频做时间轴解析
   - 识别视频中特定动作/表情/场景的"高光时刻"
   - 从长视频中截取片段
   - 对已有视频做局部增强+叠加字幕（image_edit是图片编辑，不是视频编辑）

3. "基于3小时回放提取高光片段"的核心能力是视频理解+视频剪辑——这不是doubao-creative-video的能力范畴，这是另一个完全不同的能力方向。

**判决**：🔴 案例不可行，需重新设计。当前video skill做不到"编辑已有视频"

**修正方向**：
- 如果坚持直播场景，可以改为："每天下播后，根据直播中卖得最好的3款产品，调用doubao-creative-video的UGC路线**从0生成**这3款产品的15秒带货短视频（不是从回放里剪，而是AI新拍）"
- 这个修正版不需要视频理解/剪辑能力，只需要"知道今天卖了哪几款"（数据提取）+ "为每款生成一条UGC口播视频"（video skill原生能力）
- 代价：失去了"真实直播切片"的现场感，但video skill能做到


### 案例八：大促倒计时视频矩阵

**能力动作**：21天每天早7点→算剩余天数→按阶段视觉策略→生成15秒营销视频→推群

**预判路由**：doubao-creative-video → Marketing路线 → product-marketing-ad-video-no-storyboard-ref.md

**路由匹配度**：✅ 匹配
- Marketing路线明确支持"品牌短片、广告片、信息流广告"
- 倒计时视频本质是"品牌短片+时间信息"，属于Marketing覆盖范围

**阻塞风险**：
- ⚠️⚠️ Marketing路线的确认门控：每天生成视频前都需要输出确认摘要→等用户确认。21天×每天确认=用户要确认21次。这不符合"自动化"的定位
- ⚠️ 跨天视觉一致性：Marketing路线没有"系列化视觉"的内置机制——每天生成的视频可能风格不同（今天暖调明天冷调），因为每次生成是独立调用

**建议修正**：
- 要么接受"每天需要确认一次"的半自动模式（不是全自动）
- 要么在任务配置时一次性提供21天的完整brief（每个阶段的卖点/旁白/产品）→每天自动执行时用预设参数跳过确认
- 第二个方案需要验证video skill的"确认门控"在任务模式下是否可以预设参数绕过

**判决**：⚠️ 能力匹配，但确认门控是阻塞点。需改为半自动或验证预设参数模式


### 案例九：多语种产品视频本地化 🔴🔴

1. video skill生成的是"新视频"，不是"在已有视频上改字幕/改配音"

## 案例七、九修正方案

### 案例七修正：直播后的"虚拟种草"视频自动生成

原案问题：video skill不能从3小时回放中提取高光片段

修正后：
**能力动作**：每天下播后→获取今日售出的Top3产品→为每款从0生成一条15秒UGC口播短视频

**路由**：doubao-creative-video → UGC路线 → ugc-talking-video-ref.md
- UGC路线明确支持"把一句需求变成像真人随手拍的UGC口播视频"
- 不需要视频理解/剪辑能力——完全在skill能力边界内

**修正后提示词核心**：
```
每天下午5:30触发
→ 读取今日直播销售数据（来自飞书表格"沫竹直播间·每日销售"）
→ 提取今日销量Top3产品（款号+颜色+核心卖点）
→ 对每个产品调用doubao-creative-video的UGC路线：
   生成15秒口播种草视频：
   - 主播模板每日随机抽（5个模板轮换）
   - 产品外观基于产品图库中的参考图锁定
   - 安利脚本从今日直播中的真实卖点提炼
   - 9:16竖版、保留直播原声风格
→ 命名归档+推送
```

**与原案的差异**：
- 不是"剪直播"，而是"AI新拍"——失去了现场感，但获得了可控性（主播不撞脸、产品展示更专业、不会剪到翻车片段）
- 传播逻辑变为："今天直播间最爆的三款→AI帮你生成独立的种草视频→观众看完视频去橱窗买"（短视频→橱窗的链路）

### 案例九修正：多语种产品视频独立生成

原案问题：video skill不能"基于已有视频只改字幕和配音"

修正后：
**能力动作**：每周一→基于中文产品brief，分别生成印尼/泰/英三个语种的15秒UGC带货视频（每条独立生成，画面不同但产品一致）

**路由**：doubao-creative-video → UGC路线 → ugc-talking-video-ref.md
- 关键验证点：跨语种生成时，产品外观能否保持一致（产品一致性锁的跨语言表现）

**修正后提示词核心**：
```
每周一上午10:00触发
→ 读取本周主推产品（款号+产品图+卖点）
→ 对每个语种分别调用UGC路线生成一条15秒视频：
   - 印尼语版：TikTok Shop风格、口语化、亲切
   - 泰语版：Lazada风格、有礼貌但不生硬
   - 英语版：Amazon/TikTok风格、Tech Review感
→ 产品外观基于上传的产品参考图锁定
→ 每个语种的主播自动适配当地人种和审美
→ 命名归档+推送
```

**与原案的差异**：

- 代价：三条视频画面不一样→不同市场的素材不能通用。但优势是每条都是针对当地市场的独立优化
