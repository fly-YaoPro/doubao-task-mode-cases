# Dola案例 → 豆包任务模式 × Skill 测试案例集（修订版：任务模式差异化）

> 修订日期：2026-06-30
> 修订原因：初版案例停留在"对话模式用Skill单次生图/生视频"，未体现任务模式的定时调度+上下文持久化+跨工具编排
> 方法论：五道硬筛×PC场景三维×"任务模式vs对话模式必须拉开三倍差距"


## 修订说明

**初版的根本问题**：每个案例都是"上传一张图→出一张图/一个视频"——这是对话模式就能做的事，不需要任务模式。

**任务模式的真正增量**：
1. **定时**："以后每周五下午4点你替我跑一遍"——不用自己记得
2. **上下文持久化**："品牌色、产品参数、排版偏好你记住，下次不用再说"——不用每次重输入
3. **跨工具串联**："先去后台扒数据→喂给Skill出图→推飞书群"——不用手动接力
4. **批量/规模化**："一次处理30个SKU，按模板出图"——人工不可能

**修订策略**：每个案例必须先回答"同样的需求，对话模式怎么做？任务模式怎么做？差距是多少？"——差距小于3倍不成立。


## 案例一：电商卖家·每周产品图自动处理流水线

### 过筛

| 筛 | 判断 |
|----|------|
| 第0筛（生产力） | ✅ 电商卖家经营活动 |
| 第一筛（对话能做吗） | 单次能做，但每周处理30个SKU×3张图=90次操作，对话模式要重复输入90次参数 |
| 第二筛（App能做吗） | 无App能记忆跨周模板参数自动批量处理 |
| 第三筛（大众在乎吗） | ✅ 淘宝卖家3000万，产品拍照是日常刚需 |
| PC-1 | ✅ 选图/命名/归档天然在电脑前操作 |
| PC-2 | ✅ 30 SKU×3图=90个产出物，手机完全无法管理 |
| PC-3 | ✅ 需在文件夹和浏览器之间切换 |

**对话模式 vs 任务模式差距**：30倍（1次操作 vs 90次重复操作）

### 造假背景

**商家**："沫竹Mozoo"，淘宝3皇冠女装店（虚构），店主阿琳。月均上新15-20款，每款至少需要：白底主图×1 + 搭配场景图×2 + 细节特写×1 = 4张/款 = 月均60-80张产品图。

**现在流程**：摄影师拍完→阿琳手动用美图秀秀一张张去背景调色→命名→上传淘宝后台。一款衣服处理30分钟，15款=7.5小时。每周四晚上是她雷打不动的"通宵做图夜"。

**任务模式破局**：她只要把每周拍好的原图拖进指定文件夹，任务模式自动：识别是否新品→去背景→按模板生成白底图+场景图+特写图→按款号命名→归档到对应文件夹。周四晚上她只需要抽检。

### 场景测试

**场景**：每周批量产品图处理

**提示词（任务模式配置）**：

```
帮我创建一个每周任务：

【触发条件】每周五下午2点自动执行

【任务内容】
1. 检查文件夹"E:\沫竹Mozoo\新品图待处理"，找到本周新增的产品照片
2. 对每张产品照片，使用doubao-creative-design的I2I功能做以下处理：
   - 图一·白底主图：去除原背景→替换纯白底(#FFFFFF)→产品居中→自然投影→比例1:1
   - 图二·搭配场景图：保留产品→放入简约现代试衣间/咖啡店角落场景→自然光→比例3:4
   - 图三·细节特写：自动裁切出领口、袖口、面料纹理三处细节→放大→微距质感
3. 所有输出图按"款号_图类型.png"命名（如"MZ2401_主图.png"）
4. 保存到"E:\沫竹Mozoo\产品图库\2026年6月"文件夹
5. 处理完成后生成处理报告（总数/成功/失败/耗时），推送给我

【需要记住的模板参数】
- 品牌风格：简约日系、暖调、低饱和、呼吸感留白
- 场景偏好：咖啡馆角落、现代试衣间、窗边自然光
- 不要：网红风滤镜、过饱和、浮夸大标题
- 图一经由：淘宝主图规范（1:1、白底、产品占画面85%）

【异常处理】
- 如果一张照片里有多件衣服（可能拍混了），标记为异常跳过并通知我
- 如果照片太模糊（分辨率过低），标记跳过
```

**测试关注点**：

| 关注项 | 描述 | 
|--------|------|
| 跨图产品一致性 | 同款衣服在白底图、场景图、特写图中的颜色/版型是否一致 |
| 批量命名正确性 | 款号解析是否正确（文件名→款号提取） |
| 场景不撞图 | 不同款衣服的场景图是否有变化（不是所有款都在同一个咖啡店角度） |
| 存储归档 | 输出文件夹结构是否清晰 |
| 处理报告 | 报告是否可用（像一份能发给摄影师检查的清单） |

### 场景发散

- **鞋类卖家**：批量白底图+上脚图+材质特写
- **饰品卖家**：批量白底+佩戴图+微距细节
- **食品卖家**：批量白底+食用场景+配料特写
- **数码配件**：批量白底+使用场景+接口特写


## 案例二：连锁门店·月度营销视频自动排期

### 过筛

| 筛 | 判断 |
|----|------|
| 第0筛 | ✅ 门店经营活动 |
| 第一筛 | 单次生成一个视频对话能做，但每月提前排期25号出脚本→28号出预览→1号发正式，对话记不住 |
| 第二筛 | 无App能跨月记住营销日历模板并分步骤触发 |
| PC-1 | ✅ 视频审核/品牌确认天然在电脑前 |
| PC-2 | ✅ 品牌调性文档+脚本+预览+排期表=信息密度高 |

**对话模式 vs 任务模式差距**：无法替代（对话不会主动在25号开始干活）

### 造假背景

**品牌**："唐点"，广式点心连锁（虚构，接案例七），全国12家门店。市场部1个人——小杨，负责所有门店的线上内容。

**困境**：每月都要出营销视频，流程是：想到要做→想脚本→找人拍→等剪辑→审核→发布。经常是节日前一天才赶出来，效果当然差。最崩溃的是——这事每个月都要重复一次，每次流程一模一样，但每次都要从零开始催。

### 场景测试

**提示词（任务模式配置）**：

```
帮我创建一个月度营销视频自动排期任务：

【品牌信息】（记住，以后每月复用）
- 品牌：唐点·广式点心
- 品牌调性：传统手艺+现代质感。不网红、不老气。像"舌尖"的安静，但更年轻
- 品牌色：朱砂红#C43A30、米白#FAF5F0
- 门店风格：原木+白砖墙+暖黄灯光、开放式蒸笼档口
- 视频风格偏好：15秒、竖版9:16、Marketing路线、有旁白但不快节奏、食物特写要有蒸汽和质感

【月度排期】
每月固定三个节点：

节点一：每月25号上午10点
- 执行：生成下月所有节日的营销视频清单
- 识别下月节日（节气+商业节日+周末）
- 每个节日生成一个15秒视频的脚本+分镜方案
- 推给我审核（我确认/调整脚本后再继续）

节点二：每月28号下午3点  
- 执行：基于我确认过的脚本，调用doubao-creative-video的Marketing路线
- 为每个节日生成预览视频（带水印"预览版"）
- 按节日日期排序成一张排期表（节日名称+日期+视频预览链接+发布平台）
- 推给我终审

节点三：每个节日当天早上8点
- 执行：自动生成正式版视频（去掉预览水印）
- 自动推送到飞书群"唐点·门店宣传"
- 附带发布文案（小红书版+抖音版+朋友圈版，各一段）

【本次补充信息·7月节日】
- 7月1日：香港回归纪念/下半年的仪式感
- 7月6日：小暑节气
- 7月22日：大暑节气
- 7月整月："唐点夏日点心季"品牌活动（虾饺+杨枝甘露+冰皮点心）
```

**测试关注点**：

| 关注项 | 描述 |
|--------|------|
| 跨月品牌一致性 | 每个月生成的视频是否保持品牌色调/风格/logo统一 |
| 定时触发准确 | 25号/28号/节日当天是否准时执行 |
| 脚本质量 | AI生成的节日脚本是否贴合唐点调性 |
| 审核流程 | 预览→修改→正式 的审批链路是否可用 |
| 多平台文案 | 小红书/抖音/朋友圈三版文案是否差异化 |


## 案例三：电商客服·每日FAQ热点自动生成解答海报

### 过筛

| 筛 | 判断 |
|----|------|
| 第0筛 | ✅ 客服运营 |
| 第一筛 | 对话能做一次，但做不到"每天自动统计→出图→推送"的闭环 |
| 第二筛 | 无App能自动读取客服后台+自动出图 |
| 第三筛 | ✅ 任何有客服团队的公司都有的痛点：重复问题回答效率 |
| PC-1 | ✅ 客服工作台天然PC |
| PC-2 | ✅ 问题列表+海报产出=多维度信息 |

### 造假背景

**品牌**："沫竹Mozoo"，同上女装品牌。客服团队3人，每天处理200+咨询。其中约30%是重复问题——"这件码数正吗""面料起球吗""洗了缩水吗""什么时候发货"。

**困境**：客服每天都在复制粘贴同样的话。但更浪费的是——没有一个地方把这些高频问题的答案"可视化"展示给用户。用户进店看不到，就重复问。

### 场景测试

**提示词（任务模式配置）**：

```
帮我创建一个每日客服FAQ自动处理任务：

【任务触发】每天早上8:30

【任务流程】
1. 读取昨日客服聊天记录（飞书/钉钉客服群的聊天导出）
2. 统计昨日被问最多的3个问题
3. 对每个问题：
   - 提取标准回答（从客服话术库中匹配）
   - 调用doubao-creative-design的media-creative路由，为每个问题生成一张小红书风格的"快问快答"海报
   - 海报内容：问题大字+简洁回答+产品示意图标。品牌色#米白底+#深棕字
4. 将3张海报+昨日客服数据摘要（总咨询量/高频问题Top3/平均响应时间）打包成日报
5. 推送到飞书群"沫竹·客服运营"

【需要记住的模板】
- 品牌：沫竹Mozoo，日系简约女装
- 海报风格：温暖、干净、像一张"服装护理小卡片"
- 话术风格：亲切专业、像服装顾问、不像淘宝客服
- 海报比例：1:1正方形（适合放在淘宝详情页开头+朋友圈）
- 品牌色：暖米色#FAF5F0、深棕#4A3728

【异常处理】
- 如果昨日没有新的高频问题，跳过海报生成，只推送数据日报
- 如果某个问题已有海报在库中，复用旧海报不重复生成
```

**测试关注点**：

| 关注项 | 描述 |
|--------|------|
| 问题提取准确性 | 从聊天记录中识别的Top3问题是否真的是高频问题 |
| FAQ海报复用 | 同问题一周内是否复用了已有海报（避免浪费） |
| 品牌调性保持 | 海报是否保持了品牌暖调日系风格 |
| 推送完整性 | 日报+海报的信息完整度 |
| 客服反馈可用性 | 海报能否直接用于淘宝详情页/朋友圈 |


## 案例四：多店铺运营·竞品公开定价周度对比看板

### 过筛

| 筛 | 判断 |
|----|------|
| 第0筛 | ✅ 电商经营决策 |
| 第一筛 | 对话能做一次比价，但做不到每周自动跑+自动出对比图 |
| 第二筛 | 无App能记忆竞品列表跨周对比 |
| 第三筛 | ✅ 任何多店铺运营的痛：不知道竞品什么时候调价 |
| PC-1 | ✅ 看板只能在PC屏幕上看 |
| PC-2 | ✅ 多个竞品×多个指标=高信息密度 |
| PC-3 | ✅ 需打开多个竞品店铺页面 |

### 造假背景

**运营**：小林，某天猫家电代理商运营主管。手下管3个品牌（戴森平替吸尘器、小米生态链净水器、自研加湿器），每个品牌需要盯3-5个竞品店铺。

**困境**：竞品调价永远是最后知道的。等她发现竞品降了50块，人家已经冲了一周销量。而她的"竞品监控"方式就是在浏览器收藏夹里存竞品链接，想起来就点开看一眼——想起来的频率大约是每周一次。

### 场景测试

**提示词（任务模式配置）**：

```
帮我创建一个每周竞品定价监控+对比看板任务：

【监控对象】（记住，可随时增减）
竞品A：追觅吸尘器旗舰店 - 对标我们吸尘器产品线
竞品B：石头科技官方店 - 对标我们吸尘器产品线  
竞品C：小米净水器旗舰店 - 对标我们净水器产品线
竞品D：美的净水器旗舰店 - 对标我们净水器产品线
竞品E：小熊加湿器旗舰店 - 对标我们加湿器产品线

【任务触发】每周一早上9:00

【任务流程】
1. 依次打开5个竞品店铺首页
2. 提取每个竞品的：主力SKU名称、当前售价、优惠活动（满减/优惠券）、最近一周销量（如页面可见）、上新SKU
3. 对比上周数据，标记：
   - 降价超过5% → 🔴红色预警
   - 上新款 → 🟡黄色提醒
   - 新增优惠活动 → 🟡黄色提醒
4. 调用doubao-app-builder生成PC宽屏竞品对比看板：
   - 左侧：我们产品列表+价格+利润率
   - 中间：竞品价格矩阵（横轴=竞品，纵轴=产品，格子=价格+涨跌幅）
   - 右侧：预警区（标红标黄的内容展开详情）
   - 底部：本周建议行动（"吸尘器线：追觅降了80块，建议跟降50保持价差"）
5. 保存看板为网页，推送到飞书群"电商运营·每日早会"

【需要记住的模板】
- 看板风格：深色主题、数据看板、PC宽屏（1920×1080）、专业感
- 价格敏感度：降价≥5%标红，3-5%标黄，<3%不标
- 对比逻辑：同品类同价位段竞品做同排对比
```

**测试关注点**：

| 关注项 | 描述 |
|--------|------|
| 数据提取准确性 | 从店铺页提取的价格/活动是否正确 |
| 环比计算 | 降价幅度计算是否基于上周记录 |
| 看板信息密度 | 多个竞品×多个指标是否在一屏内清晰呈现 |
| 预警有效性 | 标红/标黄是否准确（不多报不漏报） |
| 跨周记忆 | 上周价格数据是否正确保存并被本周引用 |


## 案例五：产品经理·版本发布自动生成更新公告海报

### 过筛

| 筛 | 判断 |
|----|------|
| 第0筛 | ✅ 办公场景·产品运营 |
| 第一筛 | 对话能做一次海报，但每次都要手动写文案→调排版→发布 |
| 第二筛 | 无App能自动读取发版记录+出海报 |
| 第三筛 | ✅ 所有有App/软件产品的公司都要发更新公告 |
| PC-1 | ✅ 发版公告的文案/海报排版天然在电脑端 |

### 造假背景

**产品**：某SaaS团队"云策"（虚构），做企业项目管理工具。每两周发一个版本，每次发版需要：写更新公告→做一张更新亮点海报→发到客户群+公众号+帮助中心。

**困境**：产品经理小赵每次发版最烦的不是写代码review，而是发版后的"配套物料"——明明更新内容就几点，但要排版成海报再写两套文案（客户群口语版+公众号正式版），每次耗2-3小时。发了30个版本，做了30次几乎完全一样的流程。

### 场景测试

**提示词（任务模式配置）**：

```
帮我创建一个版本发布自动公告任务：

【任务触发】每月1号和15号下午4点（我们固定的发版日）

【任务流程】
1. 读取项目管理系统（TAPD/飞书项目）中标记为"本次发布"的需求列表
2. 从需求列表中自动提取：功能名称、一句话描述、面向用户群、截图（如有）
3. 生成两类内容：
   A. doubao-creative-design出图·更新亮点海报（公众号首图）
      - 品牌色：科技蓝#2563EB+白
      - 标题："云策 V{版本号} 更新"
      - 3-5个亮点功能用卡片式布局
      - 比例：16:9（公众号封面）
   B. 两套发布文案
      - 客户群口语版：轻快、讲人话、"这次更新了三件事……"开头
      - 公众号正式版：结构化、有小标题、"本次更新包含以下内容"
4. 将海报+两套文案打包推送到飞书群"云策·产品发布"，并附上"一键复制文案"按钮

【需要记住的模板】
- 品牌：云策CloudStrategy，项目管理SaaS
- 品牌色：科技蓝#2563EB、深灰#1E293B
- 海报风格：科技感、干净、SaaS产品发布风（参考Notion/Linear的发版海报）
- 文案风格：客户群用"就像……"比喻开头，公众号用数据+结构化
- 版本号命名规则：按发版日期自动编号（如V2026.07.01）

【异常处理】
- 如果本次没有新功能发布（只有Bug修复），生成"维护更新"简化版海报
- 如果需求系统无数据可读，推送提醒让我手动输入
```

**测试关注点**：

| 关注项 | 描述 |
|--------|------|
| 需求提取准确性 | 从项目管理工具中读到的功能列表是否正确完整 |
| 海报信息密度 | 3-5个功能是否在海报上清晰呈现 |

## 案例六：人事行政·每周新员工入职欢迎物料自动生成

### 过筛

| 筛 | 判断 |
|----|------|
| 第0筛 | ✅ 企业行政/HR |
| 第一筛 | 对话能做一次海报，但每周处理不同新人信息+生成个性化物料，对话做不到持久化模板 |
| 第二筛 | 无App能记住公司品牌模板+跨周生成 |
| 第三筛 | ✅ 任何100人以上公司都有入职流程 |

### 造假背景

**HR**：某100人科技公司行政小雨。每周都有1-3个新人入职。她需要：做新人欢迎海报（发全员群）、新人名片图（发部门群）、入职指南长图。

**困境**：每周做的事完全一样——换人名+换照片+换部门+换入职日期，但就是每周都要手动操作一遍。最烦的是有时候周五忙忘了，新人周一入职，全员群里没有欢迎海报——显得公司不重视。

### 场景测试

**提示词（任务模式配置）**：

```
帮我创建一个每周新员工入职欢迎物料自动生成任务：

【任务触发】每周五下午5点（为下周一入职的新人准备物料）

【任务流程】
1. 读取飞书表格"员工入职表"中"下周入职"的所有新人信息：
   - 姓名、英文名、职位、部门、入职日期、个人照片、一句话自我介绍
2. 对每个新人，调用doubao-creative-design生成三套物料：

   物料A·全员群欢迎海报（9:16竖版）
   - 公司品牌色：深蓝#1a1a2e+亮橙#e85d04
   - 标题："欢迎{姓名}加入{公司名}"
   - 新人照片（从中部偏左）、职位+部门标签、入职日期
   - 风格：科技公司、年轻活力、不沉闷

   物料B·部门群名片图（1:1正方）
   - 简洁信息卡片：姓名、职位、一句话介绍、飞书二维码
   - 暖色背景、亲和力

   物料C·新人入职指南长图（多屏信息图）
   - Wi-Fi密码、打印机位置、食堂/周围餐厅、会议室预定、HR联系方式
   - 信息图风格、IKEA式图标指引
   - 模板固定，仅调整入职日期

3. 将所有物料按"新人姓名"分文件夹打包
4. 推送到飞书群"HR·入职准备"
5. 生成待办提醒："别忘在周一早上9点发欢迎海报到全员群"

【需要记住的模板】
- 公司名：{后续使用时补充}
- 品牌色+排版风格
- 入职指南固定内容（Wi-Fi/打印机/食堂/会议室/HR联系方式）
- 欢迎海报的设计模板

【异常处理】
- 如果入职表里没有新人照片，用姓名首字母生成一个简洁头像占位
- 如果多人同周入职，分文件夹打包
```

**测试关注点**：

| 关注项 | 描述 |
|--------|------|
| 信息提取 | 从飞书表格读取新人信息的准确性 |
| 海报个性化 | 不同新人的海报是否有个性差异 |
| 入职指南更新 | 固定模板内容是否正确填入 |
| 多新人处理 | 同周3人入职时是否每人都独立打包 |
| 推送+待办 | 物料+待办提醒的完整性 |

## 修订后总结

**初版问题**：6个案例全是"单次Skill调用" → 对话模式零差异 → 不需要任务模式

**修订后逻辑**：每个案例都是"跨周/跨月的定时自动化+多步骤编排+上下文持久化" → 对话模式完全做不到

**关键差异表**：

| 维度 | 初版（对话模式） | 修订版（任务模式） |
|------|-----------------|-------------------|
| 触发方式 | 手动上传→手动输入→手动出图 | 定时自动触发，用户只需"检查" |
| 上下文 | 每次重新描述品牌色/风格/参数 | 一次配置，跨周跨月持久化 |
| 操作次数 | 一次任务=多次手动操作 | 一次配置=永久自动运行 |
| 多步骤 | 用户自己串联"去背景→换场景→命名→归档" | 任务模式内部编排全链路 |
| 多工具 | 无 | 飞书表格读取→Skill出图→群推送 |
| PC场景 | 弱（手机也能做） | 强（看板/数据审查天然PC） |
