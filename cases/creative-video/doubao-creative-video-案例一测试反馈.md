> 测试日期：2026-06-23
> Skill版本：doubao-creative-video（含 ugc-talking-video-ref / product-marketing-ad-video-no-storyboard-ref / corporate-business-video-ref 三条路线）

## 一、场景测试反馈

### 1.1 测试场景描述

**场景**：餐饮店主每日菜单口播短视频自动生成

**具体设定**：老魏家面馆（杭州社区面馆，经营5年）的老板每天上午需要为当日主推菜品生成一条15秒口播短视频，发抖音/小红书引流。传统做法是自己架手机录制——备菜、打光、念稿、重录，一条视频折腾40分钟，且经常嘴瓢、忘词、表情僵硬。

**测试输入**（模拟任务模式每日触发）：
```
我是老魏家面馆的老板。以后每天上午10点，我会告诉你今天的推荐菜。
今天先试试：今日主推"秘制卤肉饭"，28元一份，五花肉卤了6个小时，配溏心蛋和酸菜，每天限量80份。
帮我生成一条15秒的口播短视频，竖屏，接地气的风格，像熟人推荐。
```

**Skill路由结果**：正确识别为UGC口播 → 触发 `ugc-talking-video-ref.md`

### 1.2 路由与门控行为分析

#### 一、意图识别层

Skill从用户输入中提取到的关键信号：

| 信号 | 提取值 | 路由决策 |
|------|--------|----------|
| 视频类型 | "口播短视频" | 触发UGC路线 |
| 产品信息 | "秘制卤肉饭、28元、五花肉卤6小时、溏心蛋、酸菜、限量80份" | 进入产品门控 |
| 口播风格 | "接地气"、"像熟人推荐" | 覆盖脚本引擎的tone参数 |
| 时长/比例 | "15秒"、"竖屏" | → 9:16、15s |
| 主播要求 | 未指定 | 触发主播模板随机抽卡 |

#### 二、产品门控（预期行为）

根据 UGC路线规则，skill应在看到产品信息后执行：

1. **产品图门控**：skill会首先要求用户上传产品图或确认"无参考直接生成"。如果用户上传了卤肉饭照片，进入 product_lock 流程；如果用户选择直接生成，确认摘要中标注"产品外观不保证真实一致"

2. **产品一致性锁建立**（基于用户提供的文字信息）：
   - 主料锁定：五花肉（三层分明、卤色深棕）、溏心蛋（切半、蛋黄流心状）、酸菜（碎粒、黄绿色）
   - 器皿锁定：默认白瓷碗/砂锅（餐饮口播常见）
   - 配料位置关系：米饭垫底→卤肉铺面→溏心蛋侧放→酸菜角落→葱花撒面

3. **卖点栈提取**（从用户输入自动组织）：
   ```
   核心卖点：五花肉卤制6小时（时间证据=品质）
   辅助卖点A：配溏心蛋+酸菜（组合价值）
   辅助卖点B：28元（性价比锚点）
   辅助卖点C：每天限量80份（稀缺性）
   ```

#### 三、主播模板抽卡

由于用户未指定主播，skill从5个模板中随机匹配。最可能匹配到：
- **T2 甜美邻家**（圆脸/圆眼，气质映射为"干净阳光邻家少年"）→ 最贴合"接地气街坊推荐"
- 或 **T4 中性盐系**（长脸/单眼皮，气质映射为"寡淡盐系、文艺清瘦"）→ 适合"不油腻的实在推荐"

每个主播会在 `identity_anchor` 中完整描述：模板骨骼、性别审美分支、具体发色发型、色板上衣颜色材质、场景穿搭。

#### 四、安利脚本引擎（核心差异点）

这是本案例最挑战skill能力的环节。根据UGC路线的脚本规则：

1. **Hook策略（0-2s）**：从四种hook中选一种
   - 默认优先"痛点暴击"→ "每天中午不知道吃什么的时候，这家卤肉饭我吃了三年"
   - 批次轮换时可选"数字钩子"→ "28块钱能吃到什么样的卤肉饭？"
   - 批次轮换时可选"反差钩子"→ "以前觉得社区面馆的饭都是凑数的，直到吃了这碗"

2. **安利节点结构（15s内4个节点）**：
   - 节点1（为什么现在需要）：中午犯愁吃什么/外卖吃腻了
   - 节点2（产品凭什么有效）：五花肉卤6小时，肥肉筷子一夹就化
   - 节点3（具体感受+证据）：卤汁拌饭，每一粒米都裹着汁；溏心蛋戳开跟卤汁混在一起
   - 节点4（适合谁+CTA）：住在文三路附近的真的可以来试一下，每天限量80份中午就没了

3. **去AI腔的具体手段**（这是区别于"普通对话直接写口播脚本"的关键）：
   - Speech_pace控制：15秒口播75-90字（中文5-6字/秒），避免AI式稠密输出
   - 垫词/气声融入："我跟你讲"、"你猜怎么着"、"等一下，这个卤汁…"
   - 句式禁忌：禁用排比堆叠、"不是…而是…"、"与其说…不如说…"
   - 禁止词库："绝绝子、yyds、闭眼入、无脑冲、宝藏、家人、姐妹们"
   - 推荐感强化：至少1句"强体验背书"——如"我回购第N次了"

#### 五、轻量分镜表 + 视频生成

分镜结构（15秒，约5-7个关键镜头）：
```
Shot 01 (0-2s): 主播面对镜头，表情"突然想到什么"，hook开场
Shot 02 (2-4s): 主播侧身/手引导视线，卤肉饭入画
Shot 03 (4-7s): 筷子夹起一块五花肉特写，肉颤、油脂光泽
Shot 04 (7-9s): 戳破溏心蛋，蛋黄流到卤汁里
Shot 05 (9-12s): 拌饭动作，米饭裹满卤汁
Shot 06 (12-14s): 主播面对镜头，真诚推荐表情
Shot 07 (14-15s): 产品全貌 + 店名信息 hold
```

分镜表必须包含每镜的口播文字（`{具体台词}`）、产品位置/朝向/占比、无字幕策略（UGC默认关闭字幕，除非用户明确要求花字/贴纸）。

### 1.3 区别于普通模式的突出用法

| 维度 | 普通对话模式 | 任务模式 + 创意视频Skill |
|------|-------------|--------------------------|
| **交互方式** | 每次从头描述要什么视频。第二天再来一遍："还是那家面馆，今天换卤肉饭，再做一条" | 上下文延续：skill记住了"老魏家面馆"的品牌调性、主播偏好、口播风格；第二天只需要说"今天推卤肉饭"，不用重复所有参数 |
| **内容策略** | 单条视频的文案由对话即兴生成，前后两条可能存在风格跳跃（昨天是邻家姐姐口吻，今天是美食博主腔） | 主播模板+语音音色+场景在一轮会话内保持锚定；`used_creator_profiles` 维护多日主播去重（不会连续两天用同一张脸） |
| **质量一致性** | 依赖单次对话的提示词质量，换个人可能写出完全不同的脚本 | product_lock 在会话内持久化：卤肉饭的颜色/配料/碗具绑定不变；hook_strategy 轮换确保每条视频开篇不撞车 |
| **产品安全** | 无系统性产品一致性保障——可能出现上条视频里卤肉饭是砂锅、下条变成碟子 | product_lock 机制：产品图作为最高优先级ref，所有镜头强制复述"产品数量、器型、颜色、材质、logo位置、光线方向" |
| **批量可控** | 一次一条，第二天忘了就不发了 | 任务模式可设定"每天上午10点触发"，店主只需输入今日菜品名，skill自动跑完整链路 |
| **脚本质量** | AI可能输出"绝绝子！这个卤肉饭真的yyds！" | 安利脚本引擎内置去AI腔规则：禁用网感套话、强制体感证据（"筷子一夹就化"而非"很好吃"）、语速控制75-90字/15秒 |

**最核心的差异一句话**：
> 普通对话模式 = "帮我做一条视频"；任务模式 + 创意视频Skill = "以后这家店每天的视频你帮我跑，风格一致、主播不撞脸、产品不出错、脚本不说AI话"

### 1.4 测试中暴露的能力边界与关注点

| # | 关注点 | 预期行为 | 潜在风险 |
|---|--------|----------|----------|
| 1 | 产品图缺失时的降级策略 | skill应主动要求上传产品图，或让用户确认"无参考直接生成"。如果用户选择直接生成，确认摘要必须标注"产品外观不保证真实一致" | 如果skill不问就直接生成，会出现"视频里的卤肉饭和店里实际卖的不一样"的风险——这是UGC带货最致命的信任问题 |
| 2 | 脚本去AI腔的力度 | 应出现至少一句"自然人垫词"，整体不出现排比句、不出现禁用词 | 如果skill把"去AI腔"理解为"加点语气词"，会产出"我跟你讲哦，这个卤肉饭真的绝了"——加了垫词但腔调更AI |
| 3 | 卖点排序逻辑 | 卤6小时（品质证据）应排在第一卖点；限量80份（稀缺性）应在CTA附近 | 如果skill把"28元"作为Hook开场，就走偏了——社区面馆不应该主打"便宜" |
| 4 | 主播模板与品类的匹配 | 卤肉饭是重口味、暖色系、接地气品类 → 不应匹配T1清冷高级或T3成熟御姐 | 如果抽到T1（清冷高级脸）推荐卤肉饭，会产生强烈的违和感 |
| 5 | 每日重入时的上下文保持 | 第二天说"今天推酸辣粉"时，skill应保持面馆的品牌背景、避免重新问"你是什么店" | 如果skill丢失上下文，任务模式的"连续日更"价值就坍塌了 |

## 二、场景发散

### 2.1 直接衍生场景（改品类，不改能力模式）

同一个"产品信息→口播短视频"链路，直接迁移到以下品类：

| 场景 | 人群规模 | 用例差异点 | 传播钩子 |
|------|----------|-----------|----------|
| **面包房/烘焙店** | 全国约60万家 | 产品视觉要求更高（面包的色泽、气孔、拉丝），需要更精细的product_lock；口播强调"现烤""出炉时间" | "面包师傅凌晨4点起来烤，AI 8点把视频发好了" |
| **熟食/卤味店** | 全国约100万家 | 与卤肉饭最接近，可复用大部分脚本策略；区别在"按斤卖"vs按份卖的价格表述 | "别人卤菜摊还在打字发朋友圈，AI已经帮你录好口播了" |
| **奶茶/饮品店** | 全国约50万家 | 产品是液体、杯装，product_lock需要重点关注杯型、颜色分层、顶料；口播偏"颜值+口感" | "新茶饮店一周出3款新品，AI帮你每条都配好口播视频" |
| **水果/生鲜店** | 全国约200万家 | 产品是天然形态（非包装品），product_lock挑战更大；卖点侧重"产地、当季、甜度"；口播可引入"开箱/切开"视觉 | "水果店老板娘不用自己出镜了，AI比你还懂怎么展示甜度" |
| **花店** | 全国约30万家 | 产品美学要求极高，product_lock要锁花材颜色、花型、搭配；口播偏"氛围/场景/寓意"；UGC主播模板需偏向T1或T4 | "一束花配一句文案，AI连视频带文案全搞定了" |

### 2.2 跨行业衍生场景（保留能力模式，更换信息输入源）

| 场景 | 原始模式 | 变形后 | 变形要点 |
|------|----------|--------|----------|
| **二手闲置/跳蚤市场卖家** | 餐品信息→口播 | 闲置物品描述+照片→口播介绍视频 | 产品信息结构不同（品牌、成色、原价、转让原因），但"信息→口播脚本→视频"链路相同 |
| **房产中介房源介绍** | 餐品→口播 | 房源信息→口播看房视频 | 输入从"菜品卖点"变成"户型/朝向/装修/配套"，product_lock变成room_lock（锁定户型结构）；口播语气从"街坊推荐"变成"专业客观" |
| **招聘/招工口播** | 餐品→口播 | 岗位描述→口播招聘视频 | 输入从"菜品"变成"岗位要求+薪资福利"，安利节点从"为什么需要这个菜"变成"为什么值得投这个岗位" |
| **课程/培训班推广** | 餐品→口播 | 课程大纲→口播种草视频 | 口播结构从"菜品安利"变成"学习效果安利"；product_lock变成curriculum_lock（锁定课程核心信息不出错） |

### 2.3 模式组合创新（创意视频 × 任务模式的其他能力支柱）

这是从"测试案例"向"产品能力矩阵"的推演：

**组合A：UGC口播 × 定时监控 × 跨平台适配**
```
场景：餐饮连锁品牌（如一个在5个城市有分店的品牌）
任务模式设定：每周一早上 → 监控各城市本周天气预报 → 结合天气推荐当周主推菜品 → 
为每个城市门店生成一条适配本地天气的UGC口播视频 → 9:16（抖音）/ 3:4（小红书）双版本
```
区别于单一案例的关键：引入了监控能力（天气数据）+ 多平台适配 + 多门店并行。传播钩子："五个城市的店长不用自己录视频了，AI根据天气帮你决定推什么、怎么推"

**组合B：UGC口播 × 竞品内容监控 × 差异化脚本**
```
场景：竞争激烈的餐饮品类（如火锅/烧烤）
任务模式设定：每天监控3家竞品的抖音最新视频 → 提取对方的卖点方向 → 
自动生成"错位竞争"的口播脚本（竞品打价格你打品质，竞品打环境你打口味） → 
生成口播视频
```

**组合C：Corporate路线 × 客户生命周期监控**
```
场景：B2B企业销售团队
任务模式设定：监控CRM中的客户状态变化 → 客户进入"提案阶段" → 
自动调取该客户的行业case_ref → 生成针对该客户行业的15秒案例视频 → 推给销售 → 销售发给客户
```
这是把案例一的"每日自动"逻辑和案例三的"企业视频"能力缝合到一起。

### 2.4 不适合的场景（做减法）

按照你的五道硬筛方法论，以下场景虽然技术上可能跑通，但不应作为任务模式推广案例：

| 场景 | 否决原因 | 对应筛子 |
|------|----------|----------|
| 个人Vlog日常口播 | 纯生活/娱乐，非生产力定位 | 第0筛 |
| "帮我写个口播脚本就行" | 单步任务，普通对话完全能独立完成 | 第一筛 |
| 药品/保健品口播 | 涉及功效声称+合规风险；且可能触发第4筛禁区（信息不可获取=功效无法自证） | 第四筛+合规红线 |
| 金融/理财产品口播 | 投资理财=守红线；且收益数据不可核验 | 铁律+第五筛 |

## 三、总结判断

### 3.1 案例可行性

案例一"餐饮店主每日菜单口播短视频"在能力模式五（多步链条+视频嵌入）中**可以成立**，且与案例13（房产房源视频）、案例14（电商上架素材包）的差异足够显著：

- 案例13/14的起点是"图片→视频"，能力核心是视觉素材的自动编排
- 案例一的起点是"产品文字信息→口播脚本→视频"，能力核心是安利叙事+真人感的自动化

### 3.2 建议优先推进

这个案例应该作为**任务模式案例库中能力模式五的第一个完整确证案例**（案例13/14目前标记为"已出测试提示词"未确认）。因为：
1. UGC路线是三条路线中指令最密集、约束最具体、去AI腔规则最明确的一条——测试反馈的信息密度最高
2. 餐饮是国民级行业，传播覆盖面最大
3. "每天录视频崩溃"的情绪共鸣最强

### 3.3 需要在实际测试中特别盯住的

- **脚本语言质量**：去AI腔规则写了一万多字，最终产出的脚本是"像真人朋友推荐"还是"像AI假装真人推荐"——这是定性判断，只有实际跑出来才能验证
- **产品一致性的下限**：如果用户不上传产品图就选"直接生成"，生成的卤肉饭跟实际的差多远——这决定了这个用例的"最低可接受质量"
- **跨日上下文保持**：能否在第二天、第三天持续记住面馆的品牌设定——这决定了"任务模式=以后帮我跑"这个核心价值是否成立
