# 豆包 App Builder 测试案例集（重写版）

> 生成时间：2026-06-23
> 聚焦案例：多平台电商周报数据看板
> 场景定位：PC桌面端 · 坐电脑前干活的人
> Skill核心：通过 `app_builder_agent` 将自然语言需求转化为可运行网页应用，任务模式下实现模板持久化+自动同比+异常标注

## 案例：多平台电商运营周报数据看板

### 一、案例设计

| 维度 | 内容 |
|------|------|
| **能力模式** | 新建能力模式："多源数据→周期性分析看板的自动化流水线"（任务模式三大支柱全用上：定时触发+云端浏览器操作数据源+skill调用app_builder生成看板。区别于模式二的"监控→推送消息"——本案例是"监控→分析→看板应用"，交付物是一个可视化的网页而非一条消息）|
| **目标人群** | 电商运营主管/专员、新媒体运营、渠道经理、连锁门店区域经理——典型画像：每周五下午坐在台式机或笔记本前，打开3-5个平台后台+一个Excel，花2-3小时手动扒数据、算同比、做图表、写周报结论（全国电商运营从业者约500万人，其中多平台运营占比超40%） |
| **核心痛点** | **不是"不知道数据在哪"，而是"每次把数据从四五个地方拼到一张表里太崩溃了"**。淘宝后台一个页面、京东后台一个页面、拼多多后台一个页面、抖音后台一个页面——每个平台的指标命名还不一样（淘宝叫"支付金额"、抖音叫"GMV"、拼多多叫"成交额"）。每周五下午就是一场复制粘贴+Excel公式+截图拼凑的噩梦。更烦的是，老板周一早上问"上周环比怎么样"，还得重新翻上周的Excel来对比 |
| **任务模式流程** | 首次（周五下午）：运营坐在电脑前 → 打开四个平台后台 → 口述/输入各平台本周关键数字 → app_builder生成周报看板（含数据总览+环比对比+趋势图+异常标注） → 此后每周五下午自动提醒 → 运营输入本周新数字 → 看板自动填充模板、计算环比、高亮异常 → 可导出/截图发老板 |
| **一句话传播** | "每周五下午别人在四个平台后台和Excel之间切来切去，AI帮你把周报看板自动生成了——环比涨跌一眼看到、异常指标自动标红、模板这周用完下周接着用" |
| **可量化利益** | 每周省2-3小时周报制作时间；异常指标不再漏看；环比对比不再依赖"翻上周Excel"；老板临时问数据不用慌 |

**叙事弧（30秒短视频脚本结构）：**
- 0-5s 痛苦：一个电商运营周五下午的桌面——左边屏幕开着淘宝后台、右边屏幕是京东后台、中间是Excel，来回切。镜头推到她脸上，叹了口气
- 5-12s 展示关键转折：她关掉Excel，在豆包里说"帮我做这周周报"，口述了四个平台的关键数字。屏幕上AI生成一个干净的网页看板——左边是四个平台的GMV卡片、中间是环比涨跌（红的绿的箭头一目了然）、右边是自动标注的异常（"抖音转化率跌了2个点，标红"）
- 12-20s PC场景深化：画面切到下周五，电脑右下角弹出提醒"周报时间"。她输入本周新数据。看板自动用上周模板、自动算环比、自动对比——上周还在手动画的红绿箭头，这次全自动了
- 20-25s 对比：左边是"旧世界"——四五个后台截图贴在Excel里、手动打箭头、手动写"本周总结"；右边是"新世界"——一个干净的看板，数字、图表、异常标注、对比趋势全部自动
- 25-30s 结尾："你也是那个每周五下午对着Excel崩溃的人吗？"

**过五道硬筛：**

| 筛子 | 判断 | 理由 |
|------|------|------|
| 第0筛：生产力定位 | ✅ | 电商运营=经营/搞钱场景 |
| 第一筛：普通对话能独立做吗 | ✅ 不能 | 对话可以"做一次周报"；但"记住上周模板、自动套用、自动环比、自动标注异常、跨周记忆"——对话结束后状态丢失，下周等于从零开始。只有任务模式能保持跨周上下文 |
| 第二筛：现有App能做吗 | ✅ 不能 | 市面上有各类BI工具（如简道云、帆软、Tableau）但(1)需要拖拽配置数据源和学习成本 (2)多平台数据对接需要技术开发 (3)最关键——**没有工具能做到"你说话它生成、下周自动套模板、异常自动标注"**。现有一切工具都是"配置型"而非"对话型" |
| 第三筛：大众真的在乎吗 | ✅ | 做周报是电商运营每周最烦的两件事之一（另一件是处理差评）。"周五下午周报焦虑"是500万运营人的共同记忆 |
| 第四筛：信息可获取吗 | 可探讨区 | 各平台后台数据在PC浏览器端完全可见——运营本来就开着这些页面。PC场景的优势是：浏览器操作可直接识别页面元素。初期从"运营口述/复制粘贴数字→AI生成看板"起步（安全区），成熟后可尝试"AI自动从已打开的浏览器标签页读取数据" |
| 第五筛：信息源质量 | ✅ | 数据来自各平台官方后台（淘宝千牛/京东商智/拼多多商家后台/抖音电商罗盘），均为一手官方源 |

### 四、预期Skill路由

```
用户prompt → 识别为"网页应用生成" + "数据看板/报表"
↓
信息检索（按需）：用户已提供完整数据，可跳过搜索
↓
素材组织：
  - 原始数据保留（不预计算环比——让app_builder_agent内部处理）
  - 图表需求：柱状图（GMV对比）+ 折线图（每日趋势），提供原始数据 + 图表类型建议
  - 不预生成任何图表PNG（严格遵守规则）
  - 不指定技术栈（"用ECharts画图"之类的全部不写）
↓
arch_type 判断：jspage（默认，非3D/游戏场景）
↓
user_prompt 组装结构：
  [用户原始需求]
  
  数据：
  [本周四平台完整数据表格]
  [上周四平台完整数据表格]
  
  环比计算规则：
  - 每个指标的环比 = (本周-上周)/上周 × 100%
  - 涨用绿色箭头↑，跌用红色箭头↓
  
  图表要求：
  - 柱状图：四个平台的GMV，每个平台两根柱（本周/上周），提供原始数据由app内部渲染
  - 折线图：本周七天每日GMV趋势
  
  异常标注规则：
  - 转化率环比跌幅>20%：红色高亮+警告文字
  - 退货率>6%：橙色标注
  - GMV涨幅>10%：绿色标注（正面信号）
  
  设计风格：
  - PC宽屏布局（适配1920×1080和1366×768）
  - 深色背景（类似Grafana/Datadog风格）
  - 数据卡片+表格+图表的三栏布局
  
  [🔴重要：数据结构设计为可复用模板——下周只需替换数字即可重新生成同结构看板]
↓
调用 app_builder_agent(user_prompt=完整需求, arch_type="jspage")
↓
系统卡片展示应用预览 → 用户可在PC浏览器中全屏查看
```

### 五、测试关注点（重点验证什么）

**A. 图表与数据渲染（核心能力验证）**
1. **环比自动计算**：app_builder_agent是否在页面内自动计算了环比涨跌（而非依赖用户在prompt里手动算好）。如果用户只给了本周和上周的原始数字，看板上是否显示"淘宝GMV +11.2% ↑"
2. **图表原生渲染**：柱状图和折线图是否由app_builder_agent内部渲染（不是外部生成PNG）。如果是PNG贴进去，后续"每周更新数据"就废了——没法动态刷新
3. **颜色规则执行**：涨绿色、跌红色、异常红色高亮——是否按照用户指定的规则执行
4. **数据卡片的视觉层次**：顶部总览卡片、中间表格+图表、右侧异常标注——三栏布局在宽屏下是否清晰

**B. 数据结构可复用性（任务模式核心）**
5. **模板数据结构**：看板的数据是否"硬编码"在页面里，还是设计为可替换的数据结构。这决定了"下周输入新数字→自动套模板"是否成立
6. **环比计算的自动化**：下周更新数据时，环比是否自动重新计算，还是需要人工重新输入环比数字

**C. PC端适配（场景真实性）**
7. **宽屏布局**：是否利用了宽屏空间（三栏/多栏），还是做成了手机端那样的单列堆叠——这直接决定"PC场景"的体验真实感
8. **数据密度**：PC屏幕能承载的信息量远大于手机。看板上是否充分利用了空间（表格行数、图表大小、同时可见的信息量）

**D. 后续交互能力**
9. **编辑能力**：用"按照要求修改应用+应用名称"是否能让app_builder_agent在已有看板上增加功能（如"增加一个客单价趋势图"）
10. **感知/问答（提示词D）**：针对已生成的看板问"分析一下抖音的情况"，app_builder_agent是否能基于已渲染的数据给出分析（而非"我看不到数据"）
11. **跨周更新**：下周输入新数据时，看板能否保持同样的结构、配色、布局，只换数据——这是"任务模式=以后每周用"的核心价值

**E. 边界情况**
12. **缺失数据的容错**：如果用户某周忘记提供"上周退货率"，看板是否标记为"N/A"而非显示错误或0
13. **极端数字的展示**：如果某平台环比涨了200%，数字卡片是否溢出或截断

### 六、场景发散

#### 6.1 同模式的行业迁移（改数据源，不改看板逻辑）

| 场景 | 人群 | 数据源 | 周报关注点 | PC场景特征 |
|------|------|--------|-----------|-----------|
| **新媒体运营** | 公众号/小红书/抖音/B站内容团队 | 各平台创作中心后台 | 阅读量、点赞、收藏、评论、粉丝增长、爆款内容排行 | 每周一上午分析上周内容表现，决定本周选题方向 |
| **连锁门店区域经理** | 餐饮/零售连锁的区域督导 | 各门店POS系统+外卖平台后台 | 各门店营业额、客流量、客单价、好评率、人效 | 每周五下午汇总辖区门店数据，找出垫底门店 |
| **广告投放优化师** | 信息流/搜索广告投放团队 | 巨量引擎/腾讯广告/百度营销后台 | 各渠道消耗、ROI、CPA、CTR、转化成本 | 每天上午看昨日投放数据，调整出价策略 |
| **跨境电商运营** | 亚马逊/速卖通/Shopee/Lazada卖家 | 各平台Seller Central | 各站点GMV(按汇率换算)、订单、退货、库存周转 | 每周一上午看上周各站点表现，决定补货计划 |

#### 6.2 能力模式延伸（从"周报看板"到"决策面板"）

**延伸一：周报看板 → 月度经营分析会面板**
- 人群：小微企业老板、部门负责人
- 差异：不只是"数据展示"，而是"数据+归因+建议"。看板上不仅显示环比，还自动标注"本月最大的收入增长驱动力是XX品类"、"最大的成本异常是XX"
- 任务模式：每月1号自动生成上月经营分析面板，老板在开会前收到

**延伸二：周报看板 → 竞品动态周报**
- 人群：电商运营、产品经理、市场人员
- 差异：数据源从"自己的后台"变成"竞品公开页面"。任务模式每周自动抓取竞品店铺的销量排序、价格变动、新品上架、活动变化
- 任务模式：每周一自动生成"竞品动态周报看板"——谁降价了、谁上新了、谁在做什么活动

**延伸三：单看板 → 多角色看板矩阵**
- 同一家公司：运营主管看GMV和转化率、客服主管看退货率和差评率、仓管看库存周转、老板看利润率和现金流
- 任务模式：同一套底层数据，自动生成四个不同角度的看板，分别推送给四个人

#### 6.3 不适用的场景（做减法）

| 场景 | 否决原因 | 对应筛子 |
|------|----------|----------|
| 个人记账/理财看板 | 纯生活场景，非生产力定位 | 第0筛 |
| "帮我画个柱状图" | 对话完全能独立完成的单步任务 | 第一筛 |
| 实时大屏监控（每秒刷新的那种） | app_builder是静态页面生成，不适合实时数据流 | 技术边界 |
| 股票/基金收益看板 | 投资理财守红线，且数据源涉及实时行情 | 铁律 |

### 七、与案例一（每日经营看板）的定位差异

| 维度 | 案例一：每日经营看板 | 本案例：每周周报看板 |
|------|-------------------|-------------------|
| 时间粒度 | 天 | 周 |
| 核心价值 | "不用每天手动查数据" | "不用每周手动做周报" |
| 分析深度 | 浅——今天数字对不对 | 深——这周vs上周趋势、异常归因 |
| 任务模式差异 | "每天自动更新"=省日常重复劳动 | "每周套模板+自动环比+异常标注"=省分析劳动+利用跨周记忆 |
| PC/移动倾向 | 偏移动（早上手机看一眼） | **纯PC**（坐在电脑前深度看，双屏操作） |
| 数据量 | 轻（当天的几个数） | 重（一周四个平台几十个指标+对比+趋势） |
| 典型用户动作 | 看一眼就关 | 打开看板→分析异常→写周报总结→截图/导出→发老板 |

两者不冲突——是同一个人在不同时间粒度的两个使用场景。运营每天早上用"每日看板"快速扫一眼，每周五下午用"周报看板"做深度分析和汇报材料。

## 测试执行建议

### 测试轮次设计

| 轮次 | 目的 | 提示词 | 关键验证点 |
|------|------|--------|-----------|
| 第1轮 | 基础生成：看app_builder能否理解数据看板需求 | 提示词B（轻量版） | 环比是否自动计算、图表是否原生渲染、三栏布局是否成立 |
| 第2轮 | 丰富约束：测风格/规则/异常标注的执行 | 提示词A（完整版） | 颜色规则、异常标注、数据密度、PC宽屏适配 |
| 第3轮 | 模板化：测数据结构的可复用性 | 提示词C（模板版） | 提示词中"以后每周用"的意图是否被理解并体现在数据结构中 |
| 第4轮 | 编辑：在已有看板上增改功能 | 用"按照要求修改应用"指令 | 增改是否成功、原有结构是否被破坏 |
| 第5轮 | 感知/问答：基于看板数据回答问题 | 提示词D（追问版） | app_builder_agent能否基于已渲染数据处理分析类问题 |
| 第6轮 | 跨周模拟：模拟下周更新数据 | 输入新一周数据+提示词C模板 | 看板结构是否保持、环比是否自动重算、异常标注是否更新 |

### PC端测试环境

- 建议在1920×1080或2560×1440分辨率的台式机屏幕上测试
- 打开豆包桌面端（而非手机端）
- 测试时注意：生成的看板是否在宽屏下有良好的空间利用（不是手机端那种窄单列布局）

### 特别提醒（app-builder的坑）

1. **不要预计算环比**：在user_prompt里提供本周和上周的原始数字即可，让app_builder_agent自己算环比。"淘宝本周42800 上周38500"比"淘宝环比+11.2%"更好
2. **不要预生成图表PNG**：图表必须由app_builder_agent原生渲染
3. **不要指定技术栈**：不写"用ECharts""用Chart.js""用React"
4. **编辑时用正确前缀**：对已有应用的编辑必须以"按照要求修改应用"开头
5. **想看源码走Agent路由**：不要自己用shell/cat去翻代码
